LECTURE 12 · MIDTERM 1 REVIEW + MALLOC

Returning Memory, malloc / calloc / free

Beyond the Slides — Deep Dive

How malloc actually works
malloc is NOT a system call. It's a user-space library function (in libc) that maintains a pool of memory obtained from the OS. The OS allocations come via:
  • brk / sbrk — extends the program's "data segment" (the heap break). Used for small/medium allocations.
  • mmap — maps a fresh region of pages directly. Typically used for allocations >128 KB.
Each malloc'd block has a small HEADER (typically 8-16 bytes) recording its size and free/used status. That's how free knows how much to release without you telling it. The pointer malloc returns is just past this header — so subtracting a few bytes from your pointer would let you see the size (don't actually do this).
The free list — why repeated malloc/free is fast
After you free a block, the allocator doesn't return memory to the OS — it puts the block on a "free list." The next malloc of compatible size REUSES that block. This is why malloc/free in a tight loop is fast (after the first allocation): you're just shuffling pointers in the allocator's bookkeeping. It's only slow when the allocator has to ask the OS for more, or when fragmentation forces it to walk a long free list.
Fragmentation — the hidden enemy
After many allocations of varying sizes, the heap can look like: USED-FREE(8)-USED-FREE(16)-USED-FREE(8)-USED. You have 32 bytes free total, but a request for 24 contiguous bytes FAILS. This is external fragmentation. Internal fragmentation is the wasted space inside a block (allocators round up to powers of 2 or to alignment boundaries). Long-running programs with diverse allocation patterns can degrade due to fragmentation even without leaks.
Why malloc returns void* and not int*
malloc doesn't know what you'll store. By returning void *, it commits to no specific type. C's implicit pointer conversion lets int *p = malloc(...) work without a cast. (C++ requires the cast — that's why C and C++ idioms differ here. In C, casting is considered bad form because it can hide bugs like forgetting to #include <stdlib.h>.)
malloc(0) — implementation-defined
The C standard allows malloc(0) to return either NULL or a unique non-NULL pointer that you can't dereference but you can pass to free. Don't rely on either behavior. If your code might compute a size of 0, check before calling malloc.
realloc — the third allocation function
void *realloc(void *p, size_t new_size);
Tries to resize an existing allocation. May:
  • Return the SAME pointer (block resized in place)
  • Return a NEW pointer with contents copied (old block freed)
  • Return NULL on failure (and the OLD pointer is still valid — don't lose it!)
Common bug: p = realloc(p, n) — if realloc returns NULL, you've lost the original p, leaking it. Safe pattern: tmp = realloc(p, n); if (tmp) p = tmp;

1. The Stack-Local Pointer Trap

Junior C programmers often try to "return an array" by returning a pointer to a local variable. This is broken. When the function returns, its stack frame is destroyed; the pointer now refers to memory that may be reused at any moment.

/* DO NOT DO THIS */
int *make_array() {
    int arr[10] = {0};   // stack-local
    return arr;             // returns dangling pointer!
}
Rule
NEVER return a pointer to a stack-local variable. The memory is gone the moment the function returns.

The fix: allocate on the heap with malloc. Heap memory survives until you free it.

2. malloc — manually allocate memory

void *malloc(size_t size);
  • size_t is an unsigned integer type (typically unsigned long)
  • Returns void * on success, NULL on failure
  • Memory is uninitialized — contains garbage
  • Lives on the heap until free()
#include <stdlib.h>
int *p = malloc(10 * sizeof(int));   // 10 ints
if (p == NULL) {
    fprintf(stderr, "out of memory\n");
    return -1;
}
p[0] = 42;
// ... use p ...
free(p);
p = NULL;
Idiom
Use sizeof(*p) — it stays correct if you change p's type later. int *p = malloc(N * sizeof(*p));

3. calloc — allocate AND zero

void *calloc(size_t count, size_t size);
  • Allocates count * size bytes
  • Zeros every byte (unlike malloc)
  • Two arguments: count and per-element size
int *p = calloc(10, sizeof(int));   // 10 zeroed ints

4. free — return memory to the heap

void free(void *ptr);
  • Releases the block back to the heap allocator
  • free(NULL) is safe — does nothing
  • free does NOT set the pointer to NULL — you must do it
  • Double-free = crash or corruption (undefined behavior)
  • Use-after-free = read/write to memory you don't own
free(p);
p = NULL;   // always do this

5. Memory leaks

If you malloc but never free, that memory stays allocated until the program exits. In a long-running program, leaks accumulate and the program eventually runs out of memory.

while (1) {
    int *p = malloc(1000);   // LEAK every iteration
    // ... do something ...
}   // p goes out of scope but memory stays allocated

How to approach a "make this function return a struct/array" question

Recipe
  1. Don't declare the data as a stack-local variable
  2. malloc enough bytes for the data; cast not required
  3. Check the return for NULL
  4. Initialize the contents (or use calloc to zero)
  5. Return the pointer
  6. Caller must free it later, then set their pointer to NULL

LECTURE 13 · LINKED LISTS

Singly-Linked Lists — Build, Traverse, Delete

Beyond the Slides — Deep Dive

Linked list vs array — when to choose which

In modern hardware, cache locality often dominates. An array linear scan is FASTER than a linked-list scan even though both are O(n) — because the CPU prefetches the next array element while reading the current one. Lists scatter all over memory and cause cache misses on every step.
Why use linked lists at all?
Despite the cache penalty, linked lists win when:
  • You need cheap O(1) inserts/deletes at known positions (e.g. LRU cache)
  • The collection grows unpredictably and you can't reallocate cheaply
  • You need stable pointers — adding to a list never invalidates references to existing nodes (arrays might if they realloc)
  • You're implementing other data structures (queues, stacks, hash table chains)
The Linux kernel's task list, free memory list, and many internal structures use linked lists.
Sentinel nodes — eliminating special cases
Many "list with no nodes" edge cases vanish if you keep a permanent dummy sentinel head node that contains no data. The list is "empty" when sentinel->next == NULL instead of head == NULL. Insert and delete don't need to special-case the head. Linux's circular doubly-linked lists use this technique heavily.
Why self-referential structs need pointers, not values
struct node {
    int data;
    struct node next;   // COMPILE ERROR
};
For C to compute the struct's size, it would need: size(node) = sizeof(int) + sizeof(node) — infinite recursion. A pointer is a fixed size (8 bytes on x86_64) regardless of what it points to, so struct node *next works.
Floyd's cycle detection — the "tortoise and hare"
A subtle bug: a malformed linked list with a cycle (some node's next points back into the list) makes traversal an infinite loop. Detect with two pointers, slow advancing 1 step, fast advancing 2 steps. If they ever meet, there's a cycle. If fast hits NULL first, no cycle. O(n) time, O(1) space — better than tracking visited nodes.

1. The node structure

A linked list is a chain of nodes. Each node holds a value (or values) and a pointer to the next node. The last node's next is NULL.

struct single_l {
    int data;
    struct single_l *next;   // pointer to next node
};
typedef struct single_l node_t;

Note: the struct can refer to its own type via a pointer (forward reference). It can NOT contain itself by value.

2. Allocate a node

node_t *create_node(int value) {
    node_t *n = malloc(sizeof(node_t));
    if (n == NULL) return NULL;
    n->data = value;
    n->next = NULL;
    return n;
}

3. Forward growing — append at the tail

You keep head and tail pointers. To add at the tail, you do four steps:

node_t *head = NULL;
node_t *tail = NULL;

void append(int v) {
    node_t *n = create_node(v);
    if (head == NULL) {        // 1) empty list
        head = n;
        tail = n;
    } else {
        tail->next = n;        // 2) link old tail to new node
        tail = n;              // 3) update tail
    }                          // 4) n->next is already NULL
}
The four steps
1) Create node 2) Link old tail to new node 3) Update tail to new node 4) Ensure new node's next is NULL.

4. Reverse growing — prepend at head

void prepend(int v) {
    node_t *n = create_node(v);
    n->next = head;            // 1) new node points to old head
    head = n;                  // 2) update head
    if (tail == NULL) tail = n; // 3) if list was empty
}

5. Traversal pattern

Use a "walking pointer" that starts at head and advances via p = p->next until NULL.

void print_all(node_t *head) {
    node_t *p = head;
    while (p != NULL) {
        printf("%d -> ", p->data);
        p = p->next;
    }
    printf("NULL\n");
}

6. Deletion — the "use-after-free" trap

If you free a node, you can't access its next field afterwards! Save it first.

void free_all(node_t *head) {
    node_t *p = head;
    while (p != NULL) {
        node_t *tmp = p->next;   // save next FIRST
        free(p);
        p = tmp;
    }
}
Why the temp?
After free(p), reading p->next is undefined behavior. Always save the next pointer BEFORE freeing.

7. Deletion of a specific node

To remove a target node, you need its predecessor so you can re-link.

int delete_value(node_t **head_ref, int v) {
    node_t *prev = NULL;
    node_t *curr = *head_ref;
    while (curr != NULL && curr->data != v) {
        prev = curr;
        curr = curr->next;
    }
    if (curr == NULL) return 0;     // not found
    if (prev == NULL)   *head_ref = curr->next;  // removed head
    else                prev->next = curr->next;
    free(curr);
    return 1;
}

How to approach a "linked list" question

Recipe
  1. Identify operation: insert / find / delete / traverse
  2. Use a walking pointer; check for NULL before dereferencing
  3. For insert at tail, you also need a tail pointer (or walk to end)
  4. For delete, save next BEFORE free
  5. For delete, you need the predecessor to re-link
  6. Update head/tail when removing the first/last node
  7. After free, set freed pointers to NULL

LECTURE 14 · DOUBLY-LINKED LISTS

Doubly-Linked Lists, Valgrind, the Custom CS240 Allocator

Beyond the Slides — Deep Dive

What valgrind actually does
valgrind is a dynamic binary instrumentation framework. It runs your program inside a synthetic CPU that intercepts every memory operation. Memcheck (the default tool) tracks:
  • Which bytes are allocated and which are free
  • Which bytes are initialized vs uninitialized (per-bit shadow memory!)
  • Which freed blocks have been reused
Cost: programs run 10-50x slower under valgrind. But you get exact reports of: invalid reads/writes, use-after-free, double-frees, leaks, uninitialized reads, mismatched malloc/free pairings.
Address Sanitizer — the modern alternative
Modern gcc/clang have -fsanitize=address (ASan) which is similar to valgrind but compile-time-instrumented. It's MUCH faster (2-3x slowdown vs 10-50x) and catches the same bug classes. Production-quality projects often use ASan in CI. valgrind is still better for: programs you can't recompile, kernel-style debugging, and detecting some advanced races. For CS 240, valgrind is what's available.
Reading valgrind output — what each part means
==12345== Invalid read of size 4
==12345==    at 0x4005A2: main (file.c:12)
==12345==  Address 0x520ABCD is 0 bytes after a block of size 40 alloc'd
==12345==    at 0x4C2BB8F: malloc (vg_replace_malloc.c:299)
==12345==    by 0x40058A: main (file.c:8)
  • "Invalid read of size 4" — you tried to read 4 bytes from forbidden memory
  • "at 0x... main (file.c:12)" — at line 12 of file.c (in main), the read happened
  • "is 0 bytes after a block of size 40" — you went exactly past the end of a 40-byte block
  • "alloc'd at file.c:8" — the block was malloc'd at line 8
This means: you malloc'd 40 bytes (10 ints), then read arr[10] — one past the end.
Memory leaks vs definite leaks vs possibly lost
valgrind classifies leaks into 4 categories at exit:
  • definitely lost — no pointer reaches this block (real leak, fix it)
  • indirectly lost — a leaked block's contents include pointers to other blocks
  • possibly lost — a pointer to the middle of the block exists; could be a leak or could be intentional
  • still reachable — pointers exist but you didn't free before exit; not technically a bug but bad style
The CS 240 custom malloc focuses on definite leaks and double-frees.
Why check valgrind even when your tests pass
A program can have memory bugs that don't manifest as visible failures. Reading uninitialized memory may "happen" to give the right answer this run. Buffer overruns may corrupt only memory that's about to be overwritten anyway. Then a TINY change (different optimization, different OS, different malloc implementation) suddenly exposes the bug in production. valgrind/ASan catches these latent bugs proactively.

1. Why doubly-linked?

A singly-linked list can only walk forward. To delete a node you need its predecessor — so you walk from head. Doubly-linked nodes keep a back-pointer too, allowing O(1) deletion given the node, plus reverse traversal.

struct double_l {
    int data;
    struct double_l *next_ptr;
    struct double_l *prev_ptr;
};
typedef struct double_l node_t;

2. Prepending to a doubly-linked list — order matters

void prepend(node_t **head_ref, int v) {
    node_t *n = malloc(sizeof(node_t));
    n->data = v;
    n->prev_ptr = NULL;        // 1) new node has no predecessor
    n->next_ptr = *head_ref;   // 2) point to old head
    if (*head_ref != NULL)
        (*head_ref)->prev_ptr = n; // 3) old head's prev = new node
    *head_ref = n;             // 4) head = new node
}
Four ordered steps
1) new node's prev = NULL · 2) new node's next = old head · 3) old head's prev = new node · 4) head = new node.

3. Removal from middle

Given a node n in the list, remove it in O(1):

void remove_node(node_t **head_ref, node_t *n) {
    if (n->prev_ptr) n->prev_ptr->next_ptr = n->next_ptr;
    else             *head_ref = n->next_ptr;     // was head
    if (n->next_ptr) n->next_ptr->prev_ptr = n->prev_ptr;
    free(n);
}

4. Detecting bugs — valgrind

valgrind is a tool that runs your program and reports memory errors: leaks, invalid reads/writes, double-frees, use-after-free.

$ gcc -g my_prog.c -o my_prog
$ valgrind --leak-check=full ./my_prog

Reports lines like:

How to approach a doubly-linked list operation

Recipe
  1. Draw the BEFORE state with arrows for next and prev
  2. Draw the AFTER state
  3. List the pointer rewrites needed (usually 2-4)
  4. Write them in a safe order: don't lose any node before re-linking
  5. Handle edge cases: empty list, only-node, head, tail
  6. free the removed node, set the variable to NULL

LECTURE 15 · POINTERS TO POINTERS

Double-Pointers, Internal Pointers, the Many Faces of Zero

Beyond the Slides — Deep Dive

When you actually need T**
You need a double-pointer parameter when the function must change WHICH OBJECT the caller's pointer points to. Three common cases:
  • Setting a caller's pointer to NULL — e.g., my_close(FILE **)
  • Returning a freshly-allocated pointer — e.g., create_thing(thing_t **out) instead of returning the pointer
  • Modifying the head of a linked list — prepend/delete-head changes which node IS the head
If you're only modifying what the pointer POINTS AT (not the pointer's value), T* is enough. Rule of thumb: count the asterisks needed to reach the thing you're changing.
Why use T** instead of returning T*
Two equivalent designs for "create a new linked list and give it to the caller": 
// Design A: return value
node_t *create_list(int v) {
    node_t *n = malloc(sizeof(*n));
    if (!n) return NULL;
    n->data = v; n->next = NULL;
    return n;
}

// Design B: out-parameter
int create_list(int v, node_t **out) {
    *out = malloc(sizeof(node_t));
    if (!*out) return -1;
    (*out)->data = v; (*out)->next = NULL;
    return 0;
}
Design B leaves the return value free for an error code (POSIX style). Both are common in C codebases.
Internal pointer struct = "shallow vs deep" copy problem
When a struct contains pointers, struct assignment (b = a) copies the pointer values — both structs now point to the same memory. This is a SHALLOW copy. Consequences:
  • Modifying through a->name also affects b->name
  • free(a->name) followed by free(b->name) is a double-free
  • If a->name is later reallocated to a new buffer, b->name dangles
For a true "duplicate" you need a DEEP COPY: malloc new buffers and strcpy/memcpy the contents.
qsort uses void**? Sort of
qsort's signature is void qsort(void *base, size_t n, size_t size, int (*cmp)(const void *, const void *)). Inside qsort, when it compares two elements, it passes pointers to those elements — but the elements themselves might be pointers (if you're sorting an array of strings, i.e. char **). So inside YOUR comparator, the args are const void * that you cast to char ** and dereference: strcmp(*(char**)a, *(char**)b). This is the canonical "double pointer in real code" example.
The many faces of zero — what's actually equal to what
  • 0, 0x0, 0L, 0LL, '' — all integer constants representing zero
  • 0.0 — different from integer zero in floating-point comparison contexts (and IEEE has -0.0 too)
  • NULL — typically ((void*)0); tests equal to integer 0 BUT use NULL with pointers for clarity
  • false from <stdbool.h> — defined as 0 but communicates intent
In C, integer 0 in a pointer context becomes a "null pointer constant". This is why if (p == 0), if (p == NULL), and if (!p) all work identically for pointers.

1. Why double-pointers?

To modify a caller's POINTER (not just the thing it points to), you need a pointer to the pointer. Same reason scanf takes &x for an int — but now the variable is itself a pointer.

    *fp_ptr = NULL;
}
int main() {
    FILE *fp = fopen("x", "r");
    my_close(&fp);

2. Linked-list head modification

If a function might change the head (e.g. prepend, or delete the first node), you need a node_t **head:

    node_t *n = malloc(sizeof(node_t));
    n->data = v;
    n->next = *head;     // new node points to old head
    *head = n;           // caller's head is now updated
}

// caller

3. Internal pointers — name and address fields in a struct

Sometimes a struct holds variable-length strings via internal pointers — not in-place arrays.

struct info {
    char *name;
    char *address;
};
typedef struct info info_t;

The struct itself is small (just two pointers). The actual strings live elsewhere on the heap.

Construction

    info_t *p = malloc(sizeof(info_t));
    if (p == NULL) return NULL;
    p->name    = malloc(strlen(name) + 1);
    p->address = malloc(strlen(addr) + 1);
    strcpy(p->name, name);
    strcpy(p->address, addr);
    return p;
}
Destruction with double-pointer
void delete_info(info_t **info_ptr_ptr) {
    info_t *p = *info_ptr_ptr;

    free(p->address); p->address = NULL;

LECTURE 16 · FUNCTION POINTERS + RECURSION

Function Pointers & Recursion

Beyond the Slides — Deep Dive

Function pointer syntax — read it inside out
The declaration int (*ptr)(int, int) looks weird. Read it methodically:
  1. Find the variable name: ptr
  2. Look at what's directly attached: (*ptr) — "ptr is a pointer"
  3. Then what follows: (int, int) — "...to a function taking two ints"
  4. Then the leftmost type: int — "...returning int"
Compare with int *ptr(int, int) — without parens, this is "ptr is a function returning int *". Parens around *ptr are essential.
Why is ptr(args) and (*ptr)(args) both legal?
For function pointers, dereferencing happens automatically when you call. Both forms compile and produce the same code: 
ptr(3, 4);     // implicit dereference
(*ptr)(3, 4); // explicit, K&R style
Likewise, ptr = add and ptr = &add are equivalent — function names decay to function pointers in expressions, similar to arrays.
qsort and bsearch — callbacks in the standard library
int compare(const void *a, const void *b) {
    int ia = *(const int *)a;
    int ib = *(const int *)b;
    return (ia > ib) - (ia < ib);
}
qsort(arr, n, sizeof(int), compare);
The trick "(ia > ib) - (ia < ib)" returns -1, 0, or 1 without overflow issues that return ia - ib can cause when ia and ib differ by more than INT_MAX (overflow → wrong sign).
Recursion depth — the stack overflow risk
Each recursive call adds a stack frame. The default Linux user stack is 8 MB. With ~100 bytes per frame (small function, few locals), that's ~80,000 calls deep before crash. Fibonacci, factorial, and balanced-tree traversals are fine; processing a million-node linked list recursively WILL stack-overflow. ulimit -s shows your current stack limit; you can also write recursive algorithms iteratively using your own stack on the heap.
Tail recursion — when the compiler can transform it
A "tail call" is when the recursive call is the LAST thing the function does (no work after the call returns): 
// Tail recursive — compiler may optimize to a loop
int fact_tail(int n, int acc) {
    if (n <= 1) return acc;
    return fact_tail(n - 1, acc * n);
}

// NOT tail — multiplication happens after the recursive call returns
int fact(int n) {
    if (n <= 1) return 1;
    return n * fact(n - 1);
}
With -O2, gcc can transform tail-recursive functions into loops, eliminating stack frames. C does not GUARANTEE this optimization (unlike Scheme), but it usually happens.
String literals — the read-only segment
char *s1 = "Hello";     // pointer to text/rodata segment
char s2[] = "Hello";     // stack-allocated COPY of "Hello"
For s1, "Hello" lives in the .rodata section of the executable, which is loaded into a read-only memory page. Writing through s1 causes a SIGSEGV (Linux) — not because of a NULL deref, but because of the read-only page protection. Modern compilers may even merge identical literals: two different functions writing "Hello" may share the same address.

1. Functions are addresses

Code lives at memory addresses too. You can store the address of a function in a variable — a function pointer — and call through it.

// Pointer to a function taking (int,int), returning int
int (*ptr)(int, int);

// Take its address and call
int add(int a, int b) { return a + b; }
ptr = add;          // or &add
int r = ptr(3, 4);   // or (*ptr)(3,4)
Reading the syntax
int (*ptr)(int,int) = "ptr is a pointer to a function taking (int,int) and returning int." The parens around *ptr are needed; otherwise int *ptr(int,int) means "function returning int*".

2. Generic search — pass the comparison as a parameter

node_t *list_search(node_t *head, void *target,
                    int (*cmp)(void *, void *)) {
    for (node_t *p = head; p != NULL; p = p->next) {
        if (cmp(p->data, target) == 0) return p;
    }
    return NULL;
}

// Caller supplies the comparison:
int str_cmp(void *a, void *b) {
    return strcmp((char *)a, (char *)b);
}
node_t *result = list_search(head, "target", str_cmp);

This is how qsort and bsearch in the standard library work, and it's the "callback" pattern.

3. strcmp recap

Returns 0 if equal, negative if a<b, positive if a>b. Returns int, not bool.

4. Recursion — countdown / countup

void countdown(int n) {
    if (n <= 0) return;     // base case
    printf("%d\n", n);
    countdown(n - 1);          // recursive call
}

void countup(int n) {
    if (n <= 0) return;
    countup(n - 1);
    printf("%d\n", n);        // print AFTER recursion
}

The position of the print statement (before or after the recursive call) flips the order. This is the core idea behind tree traversals.

LECTURE 17 · BINARY TREES

Binary Trees, Insert/Find, Traversal, Dynamic 2D Arrays

Beyond the Slides — Deep Dive

BST balance — why naive insertion can degenerate
A binary search tree's performance depends on its HEIGHT. A balanced tree of n nodes has height O(log n) → O(log n) operations. But naive insertion of sorted data creates a degenerate "list-like" tree of height n → O(n) operations. 
// Insert 1, 2, 3, 4 into a BST in order:
1
   2
       3
           4
// Find(4) takes 4 steps. Same as a linked list.
Self-balancing trees (AVL, red-black, B-trees) keep height O(log n) regardless of insertion order. The C++ std::map and Java TreeMap use red-black trees. CS 240's BST is the "naive" version — fine for the concept, but real code uses balanced variants.
Tree deletion — the three cases
Inserting into a BST is straightforward. Deleting is famously tricky because you might remove an internal node:
  • Leaf node (no children) — just delete
  • One child — replace deleted node with its child
  • Two children — replace with the IN-ORDER SUCCESSOR (smallest in right subtree) OR predecessor (largest in left subtree), then delete that
The third case is the tricky one: you have to find the successor (the leftmost node of the right subtree), copy its data into the slot you're "deleting," then actually remove the successor (which has at most one child).
Recursive vs iterative tree operations
Recursive tree code is shorter and clearer but uses stack proportional to tree height. For a balanced tree of n nodes, that's O(log n) — totally fine. For a degenerate BST or a linked list, recursion can stack-overflow. Iterative versions use an explicit stack on the heap (or a parent pointer in each node), trading code complexity for safety.
Row-major vs column-major — why C and Fortran differ
C lays out 2D arrays in row-major: int a[3][4] stores all 4 of row 0, then all 4 of row 1, then row 2. This is why a[i][j] = *(&a[0][0] + i*cols + j) — you skip i full rows then j elements. Fortran (and Matlab, R) use column-major: column 0, column 1, etc. This affects performance: iterate matrices in the order matching layout to maximize cache hits. In C, prefer for i { for j { a[i][j] } }. The reverse loops the same matrix but causes a cache miss every iteration on big matrices.
Two ways to allocate a dynamic 2D array — and why they differ
// Method 1: Flat (one malloc)
int *flat = malloc(rows * cols * sizeof(int));
flat[i * cols + j] = v;          // manual indexing

// Method 2: Array of pointers (rows+1 mallocs)
int **arr = malloc(rows * sizeof(int *));
for (int i = 0; i < rows; i++)
    arr[i] = malloc(cols * sizeof(int));
arr[i][j] = v;                    // natural syntax
Trade-offs:
  • Flat: one allocation, contiguous (cache-friendly), but uglier indexing
  • Array of pointers: natural syntax, allows ragged rows (different lengths), but rows are scattered → worse cache, more allocations to free
For dense numeric work, prefer flat. For text or "ragged 2D" data, prefer array of pointers.

  • File systems — directories form a tree (with hard links making it a DAG)
  • HTML/XML DOM — every web page
  • Database B-trees and B+-trees — billions of rows, log_N access
  • Compilers' abstract syntax trees

2. Create a node

3. Insert — iterative

void insert_iter(node_t **root, int v) {
    node_t *n = create_node(v);
    if (*root == NULL) { *root = n; return; }
    node_t *p = *root;
    while (1) {
        if (v <= p->data) {
            if (p->left == NULL) { p->left = n; return; }
            p = p->left;
        } else {
            if (p->right == NULL) { p->right = n; return; }
            p = p->right;
        }
    }
}

4. Insert — recursive

node_t *insert_rec(node_t *root, int v) {
    if (root == NULL) return create_node(v);
    if (v <= root->data) root->left  = insert_rec(root->left,  v);
    else                root->right = insert_rec(root->right, v);
    return root;
}
// Caller: root = insert_rec(root, value);

5. Find — recursive

node_t *tree_find(node_t *root, int v) {
    if (root == NULL || root->data == v) return root;
    if (v < root->data) return tree_find(root->left, v);
    return tree_find(root->right, v);
}

const-correctness as a contract
const isn't just for safety — it documents intent. int strlen(const char *s) tells callers: "I won't modify your string." Without const, callers can't tell whether passing a string literal (read-only) is safe. const propagates: if a function parameter is const, every function it calls with that pointer must also accept const. This is why C standard library uses const aggressively in string and memory functions.
volatile is NOT for thread synchronization
A common mistake: using volatile to share variables between threads. volatile is the wrong tool. It tells the compiler not to cache reads in registers, but it doesn't:
  • Prevent CPU reordering of memory accesses
  • Provide atomic read/write
  • Establish happens-before relationships
For threading, use _Atomic (C11), pthread mutexes, or memory barriers. volatile is for hardware registers, signal handlers, and setjmp/longjmp — places where the value can change outside the program's flow.
Reading complex const declarations — right-to-left
const int *p;             // p: pointer to const int

int *const p;             // p: const pointer to int
const int *const p;       // p: const pointer to const int
const int **q;            // q: pointer to (pointer to const int)
const int *const *r;       // r: pointer to (const pointer to const int)
Rule: const applies to whatever is IMMEDIATELY to its left, unless const is at the start of the type. Walk right-to-left, replacing * with "pointer to" and stopping at each const.

The C Preprocessor & More

Beyond the Slides — Deep Dive

The preprocessor is a separate language
The C preprocessor is not part of C the language — it's a TEXT TRANSFORMATION layer that runs first. It doesn't understand types, scope, or syntax. It only understands:
  • Lines starting with # (directives)
  • Identifier replacement (#define)
  • File inclusion (#include)
  • Conditional sections (#if/#ifdef)
This is why preprocessor errors are often confusing — the compiler sees the post-expansion text, but the error references generated code you didn't write. Use gcc -E file.c to see what the compiler actually sees.
Why macros are dangerous — multiple evaluation
#define MAX(a,b) ((a) > (b) ? (a) : (b))

int x = 5;
int y = MAX(x++, 10);   // expands to: ((x++) > (10) ? (x++) : (10))
// x is incremented TWICE if x++ > 10!
Functions evaluate arguments exactly once; macros expand textually, so any side effect in an argument runs as many times as the argument appears in the macro body. Modern code prefers static inline functions for type safety: 
static inline int max(int a, int b) {
    return a > b ? a : b;
}
But macros are unavoidable for: type-generic operations, conditional compilation, and constants used in array sizes / case labels.
Stringification # and concatenation ##
#define STR(x) #x             // turn x into a string literal
#define CONCAT(a,b) a##b      // paste tokens together

STR(hello)        // expands to "hello"
CONCAT(my_, var)  // expands to my_var

// Real use — debug print:
#define DBG(x) printf(#x " = %d
", x)
DBG(count);  // becomes: printf("count" " = %d
", count);
Adjacent string literals concatenate, so "count" " = %d " becomes "count = %d " — useful tool for debug macros.
Why "do { } while (0)" wrapping
// BAD: missing braces
#define LOG(s) printf("%s
", s); fflush(stdout)

if (verbose) LOG("hi");   // expands to:
// if (verbose) printf("%s
", "hi"); fflush(stdout);
// fflush is OUTSIDE the if!

// GOOD: do/while wrapper
#define LOG(s) do { printf("%s
", s); fflush(stdout); } while (0)

if (verbose) LOG("hi");   // works correctly
The trailing semicolon you write becomes the do-while terminator. Plain { } braces would leave a stray semicolon dangling, breaking if/else chains.
Casting truncates, doesn't round
double y = 3.7;
int i = (int) y;        // 3, NOT 4
double z = -3.7;
int j = (int) z;        // -3, NOT -4 (truncates toward zero)
For rounding, use functions in <math.h>: round(), floor(), ceil(). Casting between unrelated pointer types is also UB unless you go through char* or use a union — strict aliasing rules.
void * — the C "any pointer" but with caveats
void * is the universal pointer type. You can convert any pointer to void * and back without explicit cast (in C). But:
  • You can't dereference a void * directly — no type info
  • You can't do pointer arithmetic on void * in standard C (gcc allows it as an extension, treating it as char *)
  • Function pointers are NOT compatible with void * (different size/representation possible) — though this works on most modern systems
The standard library uses void * everywhere: malloc, memcpy, qsort callbacks. It's the C answer to "generics."

Beyond the Slides — Deep Dive

Common sources of "weird crashes"
A pattern checklist for when your C program is crashing:
  1. Stack buffer overflow — strncpy without forcing \0, scanf without width, fixed-size buffer with variable input
  2. Heap buffer overflow — wrote past end of malloc'd block
  3. Use-after-free — accessing pointer after free()
  4. Double-free — calling free twice on same pointer
  5. Uninitialized pointer — int *p; *p = 5;
  6. NULL deref — malloc returned NULL but you didn't check
  7. Off-by-one — loop condition <= n instead of < n
  8. Returning local pointer — int *f() { int x; return &x; }
  9. Stack overflow — runaway recursion or huge local array
  10. Type-punning UB — casting unrelated pointer types and dereferencing
valgrind catches 1-7. ASan and stack canaries catch 1-2 and 9. Static analyzers (clang-tidy, scan-build) catch many at compile time.
Pointer arithmetic — what's actually happening
int arr[10];
int *p = arr;
p++;
The compiler sees p++ on an int * and generates add p, sizeof(int) — typically add p, 4. The "scaling by element size" is automatic; the SAME C source produces different machine code for int * (add 4) vs long * (add 8) vs struct foo * (add sizeof(struct foo)). This is why void * arithmetic is non-standard: the compiler doesn't know what scale to use.
The "dangling else" and other parsing pitfalls
if (x) if (y) foo();
else bar();   // which if does this else attach to?
Answer: the INNER if. C uses "longest match" — else binds to the nearest unmatched if. Always brace your conditionals to remove ambiguity: 
if (x) {
    if (y) foo();
    else bar();
}
switch fall-through — feature, not bug
switch (c) {
    case 'a':
    case 'A':
        printf("letter A
");
        break;
    case 'b':
        printf("letter B
");
        // FALLTHROUGH — intentional!
    case 'c':
        printf("after b or just c
");
        break;
}
Without break, control flows into the next case. C99 and earlier silently allow this; gcc -Wimplicit-fallthrough warns. Document intentional fall-through with a comment or the C17 attribute [[fallthrough]].

ClassDefinitionExamples
Undefined behavior (UB)Anything can happen — crash, junk, or "work"Signed overflow, NULL deref, use-after-free

Beyond the Slides — Deep Dive

What the optimizer actually does
-O2 turns on dozens of optimizations. Some examples:
  • Constant folding — x = 2 + 3 becomes x = 5 at compile time
  • Dead code elimination — code with no observable effect is removed
  • Loop unrolling — replicate the body N times to reduce branch overhead
  • Inlining — replace a function call with the function's body
  • Common subexpression elimination — compute a*b once, not twice
  • Strength reduction — replace x*4 with x<<2
  • Tail call optimization — convert to a jump, no stack frame
  • Auto-vectorization — use SSE/AVX SIMD for parallel loops
  • Register allocation — keep frequently-used values in CPU registers
This is why register and many "manual optimizations" don't help with -O2: the compiler does them better. Focus your manual optimization on algorithm choice (O(n) vs O(n²)) where the compiler can't help.
When NOT to optimize
  • Don't optimize before profiling. 90% of execution time is in 10% of the code. Optimize that 10%, leave the rest readable.
  • Don't optimize debug builds. -O0 + -g gives accurate variable values in gdb. -O3 reorders code and eliminates variables, making debugging confusing.
  • Don't optimize when correctness is at stake. -O3 sometimes assumes UB doesn't happen, exposing latent bugs (signed overflow, strict aliasing).
Use gprof or perf to profile before optimizing. Real CPU bottlenecks are often surprising — what looks slow rarely is.

How shared library loading works at runtime
When you run ./prog:
  1. Kernel sees the ELF header, notices "INTERP /lib/ld-linux.so.2"
  2. Kernel runs the dynamic linker (ld-linux.so) BEFORE your program
  3. ld-linux reads the executable's "needed libraries" list (use ldd ./prog to see it)
  4. ld-linux finds each .so via LD_LIBRARY_PATH, then /etc/ld.so.cache, then default paths
  5. Each .so is mmap'd into the process's address space
  6. ld-linux resolves all "this function lives in that library" references (relocations)
  7. Then control transfers to your _start, which calls main
Run strace ./prog 2>&1 | head -50 to see this dance live. Run LD_DEBUG=libs ./prog for verbose loader output.
PIC, position-independent code, why -fPIC for shared libs
A shared library can be loaded at different addresses in different processes (mandatory for ASLR). Code with hardcoded addresses (e.g. jmp 0x40080) won't work — it's wrong if loaded at a different base. Position-independent code uses RIP-relative addressing (or a global offset table) so the same machine code works at any base address. Static libraries don't need PIC because they're copied into the executable's fixed layout.

1. The clock callback example

A simple clock struct uses volatile char fields because they're updated by an interrupt or another thread:

struct clock {
    volatile char hours;
    volatile char minutes;
    volatile char seconds;
};

void update_clock(struct clock *c) {
    c->seconds++;
    if (c->seconds >= 60) { c->seconds = 0; c->minutes++; }
    if (c->minutes >= 60) { c->minutes = 0; c->hours++; }
    if (c->hours   >= 24) { c->hours = 0; }
}

Volatile prevents the compiler from caching reads in registers — every access goes to memory.

Header guards vs #pragma once
The classic header guard: 
#ifndef MY_HEADER_H
#define MY_HEADER_H
/* declarations */
#endif
Modern alternative supported by all major compilers: 
#pragma once
/* declarations */
#pragma once is shorter, can't conflict with another file accidentally using the same macro name, and is recognized by gcc/clang/msvc. It's not in the C standard but is universally supported. CS 240 prefers the classic guards because they're guaranteed by the standard.
Why srandom(time(NULL)) is OK for class but bad in real code
time(NULL) returns seconds since the epoch — predictable, slow-changing. If two processes start within the same second, they get the same seed and the same "random" sequence. For:
  • CS 240 simulations — fine
  • Games — fine, players don't care about predictability
  • Cryptography or security — UNACCEPTABLE — use /dev/urandom or arc4random
The C standard's rand()/srand() are even weaker than random()/srandom(). For statistical work, use a proper PRNG (Mersenne Twister, PCG, xorshift).
Modulo bias in random ranges
int r = random() % 10;
If random()'s range isn't a multiple of 10, some values are slightly more likely than others. RAND_MAX is typically 2^31-1 = 2147483647. Dividing by 10 gives 214748364 with remainder 7 — values 0-7 get one more bucket than 8-9. The bias is tiny when the range is much smaller than RAND_MAX, large when ranges approach RAND_MAX. For unbiased generation, use rejection sampling: re-roll if the value falls in the "extra" range.
SDL — what's actually happening
SDL_Init opens connections to the OS's window/audio/input subsystems. SDL_SetVideoMode creates a window and a "surface" — a chunk of memory representing the framebuffer. Each "blit" (block-image-transfer) copies pixels from a source surface to a destination surface, with optional color-keying / alpha. SDL_Flip swaps two framebuffers (back buffer / front buffer) atomically, avoiding visible tearing. Modern game engines have moved to GPU APIs (OpenGL, Vulkan, DirectX, Metal) where rendering happens on the graphics card. SDL still exists as the framework that handles windowing/input/audio while delegating rendering to GPUs. CS 240's SDL is the older "blit on the CPU" model.
Game loop structure
int running = 1;
while (running) {
    // 1. Handle input events
    SDL_Event ev;
    while (SDL_PollEvent(&ev)) {
        if (ev.type == SDL_QUIT) running = 0;
    }

    // 2. Update world state (move sprites, physics)
    update_world(&state);

    // 3. Render to back buffer
    render(&state, screen);

    // 4. Flip buffers
    SDL_Flip(screen);

    // 5. Cap framerate (optional)
    SDL_Delay(16);   // ~60 FPS
}
Real games separate the update from rendering on different time bases — the world updates at a fixed rate (60 Hz) while rendering targets your monitor's rate (60/120/144 Hz). Mixing them causes physics weirdness when frame rates vary.
Why one .c per struct?
The "one struct per file" convention is the C answer to encapsulation:
  • Header (.h): the struct's TYPE declaration and the public functions that operate on it

Result: callers depend only on the header, not the implementation. You can change internal representation (e.g., switch from array to linked list) without touching callers — as long as the public interface is unchanged. This is "loose coupling" via translation units. C++'s class is essentially this convention with syntax sugar.

LECTURE 23 · BUFFER OVERFLOWS & SYSTEM CALLS

Security: Stack Smashing & The OS Boundary

Beyond the Slides — Deep Dive

Buffer overflow → control flow hijack — the classic exploit
  1. Find a function with a fixed-size local buffer and an unbounded read into it (gets, scanf without width, strcpy from external input)
  2. Send input that overflows the buffer past the saved return address
  3. Place "shellcode" (raw machine instructions) earlier in the input
  4. Overwrite the return address to point INTO your shellcode (in the buffer)
  5. When the function returns, CPU jumps to your shellcode
  6. Shellcode typically runs execve("/bin/sh", ...) — game over
This is the Morris worm (1988) and tens of thousands of CVEs since. Modern defenses make this much harder but not impossible.
The 4 modern defenses, in detail
  • Stack canaries (SSP) — gcc's -fstack-protector places a random "canary" value between locals and the return address on function entry. Before returning, the function checks the canary; if changed, abort. Defeats simple linear overflows.
  • NX / DEP / W^X — pages are either Writable or eXecutable, never both. The stack is W, not X. Shellcode placed there can't run. Defeated by ROP (return-oriented programming) which chains together small "gadgets" already in executable code.
  • ASLR — addresses of the stack, heap, libraries, and (with PIE) the executable itself are randomized each run. Attackers can't pre-compute exploit addresses. Defeated by info leaks that reveal the random offsets.
  • CFI (Control Flow Integrity) — at every indirect jump/call, check that the target is a valid function. Newer; not yet universal.
Modern attacks chain primitive bugs to bypass each defense. The exploitation game is now decades-long arms race.
User mode vs kernel mode — the privilege boundary
Modern CPUs have at least two privilege levels:
  • User mode (ring 3) — your program runs here. Can access only its own memory (via virtual address translation). Can NOT directly access hardware, kernel memory, or other processes.
  • Kernel mode (ring 0) — the OS kernel. Full hardware access.
A SYSTEM CALL is the controlled gateway between them. read() from user mode triggers a CPU instruction (syscall on x86_64) that:
  1. Saves user state
  2. Switches to kernel mode
  3. Jumps to a kernel-side handler
  4. Handler does the privileged work
  5. Returns to user mode with the result
This crossing is "expensive" — thousands of cycles vs ~1 for a function call. That's why stdio buffers reads/writes; one syscall reading 4 KB beats 4096 syscalls reading 1 byte.

System calls return -1 on error and set the global errno to a numeric error code. Inspect it:

FD leaks and file table limits
Each process has a limited number of open file descriptors — check with ulimit -n (typically 1024). Forgetting close() in a long-running server is fatal: after 1024 connections, every accept() fails with EMFILE. Worse, the kernel's file table is global; a single process leaking FDs can take down the whole system. lsof -p PID shows what FDs a process has open.

1. Why strncpy matters: buffer overflows

The classic vulnerability:

int read_login() {
    char buffer[8];          // only 8 bytes!
    gets(buffer);              // reads as much as user types
    return 0;
}

If the user types 9+ characters, gets writes past the end of buffer. On the stack, just past buffer are: saved registers, the saved frame pointer, and the return address. Overwriting the return address can hijack control flow.

2. Stack layout (32-bit, classic)

High address
┌──────────────────────────────┐
│ Function arguments (x, y)   │
├──────────────────────────────┤
│ Return address              │  ← target of overflow
├──────────────────────────────┤
│ Saved frame pointer (ebp)   │
├──────────────────────────────┤
│ Local variables (buffer[8]) │  ← buffer grows DOWN, overflow grows UP
└──────────────────────────────┘
Low address (top of stack)

The buffer grows from low addresses upward in memory; an overflow corrupts the saved frame pointer, then the return address, then the caller's local data.

3. Modern defenses

  • Stack canaries — random "canary" value placed before return address; checked on return; mismatch → abort. Default in modern gcc.
  • ASLR (Address Space Layout Randomization) — addresses change each run, so attackers can't hardcode targets
  • NX bit — stack memory is non-executable

char buffer[8];

strncpy(buffer, src, sizeof(buffer));
buffer[sizeof(buffer) - 1] = '\0';

Reading a stack dump — putting it all together
Suppose gdb shows the stack as bytes: 
0x7fff0040: 41 41 41 41 41 41 41 41   ← buffer[0..7]
0x7fff0048: 41 41 41 41 41 41 41 41   ← buffer[8..15]
0x7fff0050: 00 00 00 00 00 00 00 00   ← saved frame pointer
0x7fff0058: a0 04 40 00 00 00 00 00   ← return address (jumps to 0x4004a0)
0x7fff0060: 0a 00 00 00 00 00 00 00   ← argument: int n = 10
0x7fff0068: ...                       ← caller's frame
Stack grows DOWN, so lower addresses = newer (locals) and higher = older (caller's frame, args). 0x41 = 'A' — the function got input "AAAAAAAA...". Notice all 16 bytes of buffer are 'A's, which means the input was ≥ 16 chars. Now look at frame pointer and return address — if they're still intact, no overflow yet. If 0x7fff0050 contained more 0x41's, the saved frame pointer was overwritten — overflow detected. If 0x7fff0058 had 0x41's, the return address itself was corrupted — full hijack potential.
x86 calling conventions — where args go
Architecture / ABIFirst int args go inStack used for
x86 (cdecl, 32-bit)Stack onlyAll args, in reverse order
x86_64 System V (Linux/macOS)rdi, rsi, rdx, rcx, r8, r9Args 7+, large structs
x86_64 Microsoft (Windows)rcx, rdx, r8, r9Args 5+, plus shadow space

Float arguments use separate registers (xmm0-xmm7 on x86_64 SysV). The return value goes in rax (rax+rdx for 128-bit returns). This is why a 32-bit stack overflow exploit looks different from 64-bit: the first args aren't even on the stack.
The frame pointer — what it's actually for
On x86_64, the frame pointer (rbp) holds the START of the current function's stack frame. Locals are accessed at fixed offsets from rbp: [rbp-4] = local x, [rbp-8] = local y. This makes debugging easier — gdb walks the frame pointers to print the call stack. Modern -O2 omits the frame pointer (-fomit-frame-pointer is implicit), using rbp as a general register and accessing locals via rsp-relative addressing. This frees one register for general use but makes manual stack walking harder. Recent Linux kernels have been re-enabling frame pointers for better profiling/tracing.
Core dumps — what's actually in them
A core file is essentially the process's entire memory at crash time, in ELF format. Includes:
  • All writable memory regions (stack, heap, .data, .bss)
  • Register state at the moment of crash
  • Memory map (which segments came from which library)
Excludes:
  • Code segments (regenerable from the executable file)
  • Mapped read-only files
Open with gdb prog core and you can inspect every variable, the stack at every level, and use print *p as if the program were paused. Core files are large (~size of the heap), which is why ulimit -c defaults to 0 in many distros.
When goto isn't evil — the cleanup pattern
The "Linux kernel" cleanup pattern is the classic legitimate use: 
int setup() {
    int rc = 0;
    char *buf1 = malloc(100);
    if (!buf1) { rc = -1; goto err1; }
    char *buf2 = malloc(200);
    if (!buf2) { rc = -1; goto err2; }
    FILE *fp = fopen("x", "r");
    if (!fp)   { rc = -1; goto err3; }

    /* do work */
    fclose(fp);
err3: free(buf2);
err2: free(buf1);
err1: return rc;
}
Without goto, you'd nest if-else 3 levels deep or duplicate the cleanup code. The labels-and-goto pattern is the cleanest expression of "release everything I've acquired so far, in reverse order." This is essentially RAII (C++) or defer (Go) before those existed.

1. Stack diagram (revisited) — the takehome quiz pattern

You'll be shown a function and asked to identify which line of memory holds the local variable, which holds the return address, which holds the argument. The mental layout (32-bit, growing downward in address):

High address  ┌─────────────────────────────┐
              │  argument 2 (e.g. y)        │
              │  argument 1 (e.g. x)        │
              ├─────────────────────────────┤
              │  RETURN ADDRESS             │  ← caller's next instr
              ├─────────────────────────────┤
              │  saved frame pointer (ebp)  │
              ├─────────────────────────────┤
              │  local variables            │  ← e.g. char buf[8]
Low address   └─────────────────────────────┘  (top of stack)

The compiler typically reserves locals first, then the return address sits above them. A buffer overflow inside buf writes upward (toward higher addresses) and reaches the saved fp, then the return address.

2. The walk() example

void walk(int x, int y) {
    char buf[16];
    /* ... */
}
int main() {
    walk(3, 7);
    return 0;
}

Inside walk: the stack (high → low) holds y=7, x=3, return_addr_into_main, saved_ebp, then 16 bytes of buf. Writing past buf corrupts saved_ebp, then return_addr.

4. The goto statement

Dijkstra's 1968 paper "Go To Statement Considered Harmful" cemented goto's reputation. C still has it.

int work() {
    int *a = malloc(...);
    int *b = malloc(...);
    int *c = malloc(...);
    if (a == NULL) goto err;
    if (b == NULL) goto err;
    if (c == NULL) goto err;
    /* normal path */
    return 0;
err:
    free(a); free(b); free(c);
    return -1;
}
When goto IS reasonable
  • Breaking out of deeply nested loops
  • Cleanup-on-error idiom (Linux kernel uses this everywhere)
  • State machines with explicit transitions

Outside these patterns, prefer structured control flow.

make and the Build System

Beyond the Slides — Deep Dive

Why the TAB requirement is famously annoying
make's parser was written in 1976. The author (Stuart Feldman) said:
"I realized this was a bad idea, but the next morning I had a dozen users."
By the time the bug was obvious, too many Makefiles relied on it. The TAB-vs-spaces issue has caused millions of hours of confusion. Modern alternatives (Ninja, Bazel, Meson) avoid this. But make is universal, so we live with it. Some editors auto-convert tabs to spaces — disable that for Makefiles.
Phony targets — why they matter
# BAD - if a file named "clean" exists, make thinks the target is up-to-date
clean:
	rm -f *.o my_prog

# GOOD - .PHONY tells make to ALWAYS run this
.PHONY: clean
clean:
	rm -f *.o my_prog
Common phony targets: all, clean, install, test, distclean, dist. Without .PHONY, anyone naive can break the build by creating a file with that name.
Make's variable flavors
A = $(B)        # RECURSIVELY expanded — re-evaluated each use
A := $(B)       # SIMPLY expanded — evaluated NOW, frozen
A ?= value      # SET only if A isn't already defined
A += extra      # APPEND to existing

CFLAGS = -Wall
CFLAGS = $(CFLAGS) -g    # INFINITE RECURSION!

CFLAGS := -Wall
CFLAGS := $(CFLAGS) -g   # works: -Wall -g
Use := when you don't want late binding surprises.
make -j parallelism — why it sometimes fails
make -j8 runs up to 8 recipes in parallel — much faster on multicore machines. But:
  • Recipes that don't declare ALL their dependencies will fail when they run before what they actually need
  • Recipes writing to the same file race each other
  • Order of error messages becomes unpredictable
A correct Makefile works at any -j level. A "works only with -j1" Makefile has incomplete dependencies. You can find these bugs by adding .NOTPARALLEL: as a debugging tool.
Auto-generated dependencies — the missing piece
A naive Makefile only knows: "main.o depends on main.c". But what about the .h files main.c includes? If you change a header, dependent .c files need rebuilding too. Solution: gcc -MMD: 
CFLAGS = -Wall -MMD
%.o: %.c
	$(CC) $(CFLAGS) -c $< -o $@

-include $(OBJS:.o=.d)
gcc generates a .d file alongside each .o, listing every header included. The Makefile -includes them. Now changing any included header triggers rebuild of every .c that includes it, transitively. This is what real projects do — without it, "make clean && make" is the only safe rebuild.

LECTURE 26 · INTERFACING WITH HARDWARE

Ports, Linux /dev, Raspberry Pi, GPIO

Beyond the Slides — Deep Dive

Memory-mapped I/O — how software talks to hardware
On the Raspberry Pi (and most ARM SoCs), GPIO pins are controlled by writing to specific MEMORY ADDRESSES. The CPU's memory subsystem routes accesses to those addresses to the GPIO peripheral instead of RAM. From C: 
#define GPIO_BASE 0x3F200000   // physical address on Pi 3
volatile unsigned int *gpio = (unsigned int *) GPIO_BASE;
gpio[0x1C / 4] = 1 << 17;   // Set GPIO pin 17 high
The volatile is essential — the compiler must NOT cache reads or eliminate writes. On Linux user-space, you can't directly access these physical addresses; you mmap /dev/mem first.
Why peripherals require careful timing
GPIO writes are FAST (nanoseconds). But hardware response is often SLOW (microseconds for I2C, milliseconds for relays). The CPU can issue thousands of writes per millisecond, faster than the hardware can respond. Common patterns:
  • Polling — repeatedly read a status register until it shows "done"
  • Delays — sleep for a known time before reading
  • Interrupts — hardware tells the CPU when it's ready
For real-time work, you need to know your hardware's timing. Datasheets specify minimum hold/setup times, propagation delays, and clock requirements.
GPIO direction registers
A GPIO pin is configurable: input (read voltage) or output (drive voltage). Modern SoCs have separate registers:
  • FSEL (function select) — chooses pin role: input, output, or alternate function (UART, SPI, etc.)
  • SET — write a 1 to drive the corresponding pin HIGH
  • CLR — write a 1 to drive the corresponding pin LOW
  • LEV — read to get current pin state
Notice SET and CLR are write-only — separate registers avoid race conditions where one bit is being toggled while reading the rest.
Why level shifters matter
Many sensors use 5V logic (Arduino-era hardware) but the Pi's GPIOs are 3.3V tolerant only. Connecting 5V to a 3.3V GPIO without a level shifter:
  • BEST CASE: the input clamping diodes conduct, drawing excess current that damages the SoC over time
  • WORST CASE: instant chip death
Solutions: voltage divider (resistors), bidirectional level shifter chips, or sensors that support 3.3V mode. Always check pin voltage in the datasheet.
The /dev abstraction — UNIX's "everything is a file"
UNIX's design philosophy: hide hardware details behind file-like APIs. open, read, write, close work on:
  • Regular files
  • Pipes
  • Network sockets
  • Devices (/dev/ttyUSB0, /dev/sda)
  • Special files (/dev/null, /dev/zero, /dev/random)
  • Pseudo-files (/proc/cpuinfo, /sys/class/gpio)
This unification makes programs simpler — a tool that processes files automatically works on devices and sockets. cat works on serial ports, network connections, and files alike.

Real networking is layered, each layer adding a feature on top of the one below:

TCP gives you reliability for free but pays in latency. UDP is faster but the application must handle losses — sometimes acceptable (a dropped video frame is fine; the next one comes in 16ms).
The 3-way handshake — how TCP starts a connection
Client                    Server
  ----- SYN seq=A ----->
  <-- SYN-ACK seq=B,A+1 --
  ----- ACK A+1,B+1 --->
  <==== Connection established ====>
This exchange establishes initial sequence numbers (random for security) and confirms both directions work. After this, every TCP segment carries a sequence number; the receiver acks back so the sender knows it arrived. Lost packets are detected by missing acks (after a timeout) and retransmitted.
Network byte order — big-endian everywhere
The internet uses big-endian for all multi-byte values in protocol headers. x86 is little-endian. Hence htons (host-to-network short), htonl (long), ntohs, ntohl — they swap bytes if needed. On a big-endian machine they're no-ops; on x86 they actually swap. Forgetting htons is the #1 socket-programming bug — the port number ends up correct on a big-endian machine, wrong everywhere else.
Why blocking calls block — accept, recv, connect
By default, sockets are BLOCKING:
  • accept() blocks until a client connects
  • recv() blocks until at least 1 byte arrives
  • connect() blocks until the handshake completes (or times out)
This is a problem for servers: while you're blocked in recv() with one client, others can't be served. Three solutions:
  • Threads — one per connection (or thread pool)
  • fork() — one process per connection (Apache prefork)
  • Non-blocking + epoll/kqueue — single thread juggling thousands of connections (nginx, redis)
Modern high-performance servers use the epoll approach. Lower-overhead, scales to 10,000+ connections per thread.
DNS resolution under the hood
"Look up www.purdue.edu" actually involves:
  1. Browser checks its own cache
  2. OS asks the configured resolver (typically your ISP's DNS)
  3. Resolver checks ITS cache
  4. If not cached: resolver asks a root server "where's .edu?"
  5. Root server: "ask a-l.edu-servers.net"
  6. Resolver asks an .edu server: "where's purdue.edu?"
  7. .edu server: "ask ns1.purdue.edu"
  8. Resolver asks ns1.purdue.edu: "what's www.purdue.edu?"
  9. Authoritative answer: "128.10.5.32"
  10. Resolver caches and returns to OS
  11. OS caches and returns to browser
  12. Browser opens TCP connection to 128.10.5.32:443
The whole chain typically takes 10-100ms but is cached for hours/days. dig www.purdue.edu on Linux shows the trace.
Sockets are file descriptors
On UNIX, a socket is just an int (a file descriptor) — same type as those from open(). You can use read()/write() on sockets, or the more powerful recv()/send() with flags. You can select()/poll() on a mix of sockets and files. This unification means cat works on a TCP connection, tail -f works on a network log, etc. Windows separated networking from files (winsock) — UNIX kept them unified.
TLS / HTTPS — the layer above TCP
HTTPS is HTTP over TLS over TCP. TLS adds:
  • Encryption — eavesdroppers see only ciphertext
  • Authentication — certificates prove the server's identity
  • Integrity — tampering is detectable
The handshake (after TCP is established) negotiates a cipher suite, exchanges keys, and verifies certificates. After handshake, the application sees a "secure stream" similar to a normal TCP stream. Real apps use libraries like OpenSSL — implementing TLS yourself is suicide.

1. Bubble sort — a simple sort

void bubble(int *a, int n) {
    for (int i = 0; i < n - 1; i++) {
        for (int j = 0; j < n - 1 - i; j++) {
            if (a[j] > a[j + 1]) {
                int t = a[j]; a[j] = a[j+1]; a[j+1] = t;
            }
        }
    }
}
XOR swap (no temp)
a[j]   ^= a[j+1];
a[j+1] ^= a[j];
a[j]   ^= a[j+1];   // careful: doesn't work if same address
Improved with early exit
void bubble2(int *a, int n) {
    int swapped = 1;
    while (swapped) {
        swapped = 0;
        for (int j = 0; j < n - 1; j++) {
            if (a[j] > a[j+1]) {
                int t = a[j]; a[j] = a[j+1]; a[j+1] = t;
                swapped = 1;
            }
        }
    }
}

Bubble sort is O(n²) — instructive but not efficient. Real code uses qsort.

2. Networking basics

Three problems to solve:

  1. Find the system — IP addressing, DNS
  2. Route to it — routers, hops
  3. Speak the protocol — TCP, UDP, application protocols (HTTP, etc.)

3. IP addressing

  • IPv4: 32 bits, written nnn.nnn.nnn.nnn (e.g. 128.10.116.31)
  • IPv6: 128 bits
  • Public addresses: globally routable; private: 10.x, 172.16-31.x, 192.168.x
  • NAT (Network Address Translation): one public IP fronts many private IPs
  • IP is unreliable — no acks, no ordering, no delivery guarantee

4. DNS — Domain Name System

  • Translates names like purdue.edu to IP addresses
  • Hierarchical: root servers → .edu authoritative → purdue.edu authoritative
  • Heavily cached for speed
  • Reverse DNS: IP → name

5. TCP — reliable transport on top of IP

  • Adds: acknowledgements, retransmission, ordering, flow control, congestion control
  • A "host" is identified by IP + port (16-bit)
  • Ports below 1024 are privileged (require admin to bind)
  • UDP is the unreliable alternative — less overhead, no ordering/retransmit

6. Clients vs servers

  • Server: waits passively, accepts connections
  • Client: actively initiates a connection
  • A program can be both at different times

7. Sockets — the API

A socket is one endpoint of a connection. To the OS it's a file descriptor; you can read/write it like a file. Specialized socket calls: recv/send.

Server steps
int sfd = socket(AF_INET, SOCK_STREAM, 0);
struct sockaddr_in addr = {0};
addr.sin_family      = AF_INET;
addr.sin_port        = htons(8080);
addr.sin_addr.s_addr = INADDR_ANY;

listen(sfd, 5);                     // 5 = backlog

send(cfd, response, len, 0);
close(cfd);
close(sfd);

socket → bind → listen → accept → recv/send → close.
