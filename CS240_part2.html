LECTURE 12 · MIDTERM 1 REVIEW + MALLOC

Returning Memory, malloc / calloc / free

Beyond the Slides — Deep Dive

How malloc actually works
malloc is NOT a system call. It's a user-space library function (in libc) that maintains a pool of memory obtained from the OS. The OS allocations come via:
  • brk / sbrk — extends the program's "data segment" (the heap break). Used for small/medium allocations.
  • mmap — maps a fresh region of pages directly. Typically used for allocations >128 KB.
Each malloc'd block has a small HEADER (typically 8-16 bytes) recording its size and free/used status. That's how free knows how much to release without you telling it. The pointer malloc returns is just past this header — so subtracting a few bytes from your pointer would let you see the size (don't actually do this).
The free list — why repeated malloc/free is fast
After you free a block, the allocator doesn't return memory to the OS — it puts the block on a "free list." The next malloc of compatible size REUSES that block. This is why malloc/free in a tight loop is fast (after the first allocation): you're just shuffling pointers in the allocator's bookkeeping. It's only slow when the allocator has to ask the OS for more, or when fragmentation forces it to walk a long free list.
Fragmentation — the hidden enemy
After many allocations of varying sizes, the heap can look like: USED-FREE(8)-USED-FREE(16)-USED-FREE(8)-USED. You have 32 bytes free total, but a request for 24 contiguous bytes FAILS. This is external fragmentation. Internal fragmentation is the wasted space inside a block (allocators round up to powers of 2 or to alignment boundaries). Long-running programs with diverse allocation patterns can degrade due to fragmentation even without leaks.
Why malloc returns void* and not int*
malloc doesn't know what you'll store. By returning void *, it commits to no specific type. C's implicit pointer conversion lets int *p = malloc(...) work without a cast. (C++ requires the cast — that's why C and C++ idioms differ here. In C, casting is considered bad form because it can hide bugs like forgetting to #include <stdlib.h>.)
malloc(0) — implementation-defined
The C standard allows malloc(0) to return either NULL or a unique non-NULL pointer that you can't dereference but you can pass to free. Don't rely on either behavior. If your code might compute a size of 0, check before calling malloc.
realloc — the third allocation function
void *realloc(void *p, size_t new_size);
Tries to resize an existing allocation. May:
  • Return the SAME pointer (block resized in place)
  • Return a NEW pointer with contents copied (old block freed)
  • Return NULL on failure (and the OLD pointer is still valid — don't lose it!)
Common bug: p = realloc(p, n) — if realloc returns NULL, you've lost the original p, leaking it. Safe pattern: tmp = realloc(p, n); if (tmp) p = tmp;

1. The Stack-Local Pointer Trap

Junior C programmers often try to "return an array" by returning a pointer to a local variable. This is broken. When the function returns, its stack frame is destroyed; the pointer now refers to memory that may be reused at any moment.

/* DO NOT DO THIS */
int *make_array() {
    int arr[10] = {0};   // stack-local
    return arr;             // returns dangling pointer!
}
Rule
NEVER return a pointer to a stack-local variable. The memory is gone the moment the function returns.

The fix: allocate on the heap with malloc. Heap memory survives until you free it.

2. malloc — manually allocate memory

void *malloc(size_t size);
  • size_t is an unsigned integer type (typically unsigned long)
  • Returns void * on success, NULL on failure
  • Memory is uninitialized — contains garbage
  • Lives on the heap until free()
#include <stdlib.h>
int *p = malloc(10 * sizeof(int));   // 10 ints
if (p == NULL) {
    fprintf(stderr, "out of memory\n");
    return -1;
}
p[0] = 42;
// ... use p ...
free(p);
p = NULL;
Idiom
Use sizeof(*p) — it stays correct if you change p's type later. int *p = malloc(N * sizeof(*p));

3. calloc — allocate AND zero

void *calloc(size_t count, size_t size);
  • Allocates count * size bytes
  • Zeros every byte (unlike malloc)
  • Two arguments: count and per-element size
int *p = calloc(10, sizeof(int));   // 10 zeroed ints

4. free — return memory to the heap

void free(void *ptr);
  • Releases the block back to the heap allocator
  • free(NULL) is safe — does nothing
  • free does NOT set the pointer to NULL — you must do it
  • Double-free = crash or corruption (undefined behavior)
  • Use-after-free = read/write to memory you don't own
free(p);
p = NULL;   // always do this

5. Memory leaks

If you malloc but never free, that memory stays allocated until the program exits. In a long-running program, leaks accumulate and the program eventually runs out of memory.

while (1) {
    int *p = malloc(1000);   // LEAK every iteration
    // ... do something ...
}   // p goes out of scope but memory stays allocated

How to approach a "make this function return a struct/array" question

Recipe
  1. Don't declare the data as a stack-local variable
  2. malloc enough bytes for the data; cast not required
  3. Check the return for NULL
  4. Initialize the contents (or use calloc to zero)
  5. Return the pointer
  6. Caller must free it later, then set their pointer to NULL

LECTURE 13 · LINKED LISTS

Singly-Linked Lists — Build, Traverse, Delete

Beyond the Slides — Deep Dive

Linked list vs array — when to choose which
OperationArrayLinked List
Random access (arr[i])O(1)O(n)
Insert at frontO(n) — shift everythingO(1)
Insert at end (with tail)O(1) amortizedO(1)
Insert in middleO(n) — shiftO(1) given the node, O(n) to find
DeleteO(n)O(1) given prev pointer
Cache localityExcellent (contiguous)Terrible (scattered)
Memory per elementJust dataData + pointer (8 bytes overhead each)
In modern hardware, cache locality often dominates. An array linear scan is FASTER than a linked-list scan even though both are O(n) — because the CPU prefetches the next array element while reading the current one. Lists scatter all over memory and cause cache misses on every step.
Why use linked lists at all?
Despite the cache penalty, linked lists win when:
  • You need cheap O(1) inserts/deletes at known positions (e.g. LRU cache)
  • The collection grows unpredictably and you can't reallocate cheaply
  • You need stable pointers — adding to a list never invalidates references to existing nodes (arrays might if they realloc)
  • You're implementing other data structures (queues, stacks, hash table chains)
The Linux kernel's task list, free memory list, and many internal structures use linked lists.
Sentinel nodes — eliminating special cases
Many "list with no nodes" edge cases vanish if you keep a permanent dummy sentinel head node that contains no data. The list is "empty" when sentinel->next == NULL instead of head == NULL. Insert and delete don't need to special-case the head. Linux's circular doubly-linked lists use this technique heavily.
Why self-referential structs need pointers, not values
struct node {
    int data;
    struct node next;   // COMPILE ERROR
};
For C to compute the struct's size, it would need: size(node) = sizeof(int) + sizeof(node) — infinite recursion. A pointer is a fixed size (8 bytes on x86_64) regardless of what it points to, so struct node *next works.
Floyd's cycle detection — the "tortoise and hare"
A subtle bug: a malformed linked list with a cycle (some node's next points back into the list) makes traversal an infinite loop. Detect with two pointers, slow advancing 1 step, fast advancing 2 steps. If they ever meet, there's a cycle. If fast hits NULL first, no cycle. O(n) time, O(1) space — better than tracking visited nodes.

1. The node structure

A linked list is a chain of nodes. Each node holds a value (or values) and a pointer to the next node. The last node's next is NULL.

struct single_l {
    int data;
    struct single_l *next;   // pointer to next node
};
typedef struct single_l node_t;

Note: the struct can refer to its own type via a pointer (forward reference). It can NOT contain itself by value.

2. Allocate a node

node_t *create_node(int value) {
    node_t *n = malloc(sizeof(node_t));
    if (n == NULL) return NULL;
    n->data = value;
    n->next = NULL;
    return n;
}

3. Forward growing — append at the tail

You keep head and tail pointers. To add at the tail, you do four steps:

node_t *head = NULL;
node_t *tail = NULL;

void append(int v) {
    node_t *n = create_node(v);
    if (head == NULL) {        // 1) empty list
        head = n;
        tail = n;
    } else {
        tail->next = n;        // 2) link old tail to new node
        tail = n;              // 3) update tail
    }                          // 4) n->next is already NULL
}
The four steps
1) Create node 2) Link old tail to new node 3) Update tail to new node 4) Ensure new node's next is NULL.

4. Reverse growing — prepend at head

void prepend(int v) {
    node_t *n = create_node(v);
    n->next = head;            // 1) new node points to old head
    head = n;                  // 2) update head
    if (tail == NULL) tail = n; // 3) if list was empty
}

5. Traversal pattern

Use a "walking pointer" that starts at head and advances via p = p->next until NULL.

void print_all(node_t *head) {
    node_t *p = head;
    while (p != NULL) {
        printf("%d -> ", p->data);
        p = p->next;
    }
    printf("NULL\n");
}

6. Deletion — the "use-after-free" trap

If you free a node, you can't access its next field afterwards! Save it first.

void free_all(node_t *head) {
    node_t *p = head;
    while (p != NULL) {
        node_t *tmp = p->next;   // save next FIRST
        free(p);
        p = tmp;
    }
}
Why the temp?
After free(p), reading p->next is undefined behavior. Always save the next pointer BEFORE freeing.

7. Deletion of a specific node

To remove a target node, you need its predecessor so you can re-link.

int delete_value(node_t **head_ref, int v) {
    node_t *prev = NULL;
    node_t *curr = *head_ref;
    while (curr != NULL && curr->data != v) {
        prev = curr;
        curr = curr->next;
    }
    if (curr == NULL) return 0;     // not found
    if (prev == NULL)   *head_ref = curr->next;  // removed head
    else                prev->next = curr->next;
    free(curr);
    return 1;
}

How to approach a "linked list" question

Recipe
  1. Identify operation: insert / find / delete / traverse
  2. Use a walking pointer; check for NULL before dereferencing
  3. For insert at tail, you also need a tail pointer (or walk to end)
  4. For delete, save next BEFORE free
  5. For delete, you need the predecessor to re-link
  6. Update head/tail when removing the first/last node
  7. After free, set freed pointers to NULL

LECTURE 14 · DOUBLY-LINKED LISTS

Doubly-Linked Lists, Valgrind, the Custom CS240 Allocator

Beyond the Slides — Deep Dive

What valgrind actually does
valgrind is a dynamic binary instrumentation framework. It runs your program inside a synthetic CPU that intercepts every memory operation. Memcheck (the default tool) tracks:
  • Which bytes are allocated and which are free
  • Which bytes are initialized vs uninitialized (per-bit shadow memory!)
  • Which freed blocks have been reused
Cost: programs run 10-50x slower under valgrind. But you get exact reports of: invalid reads/writes, use-after-free, double-frees, leaks, uninitialized reads, mismatched malloc/free pairings.
Address Sanitizer — the modern alternative
Modern gcc/clang have -fsanitize=address (ASan) which is similar to valgrind but compile-time-instrumented. It's MUCH faster (2-3x slowdown vs 10-50x) and catches the same bug classes. Production-quality projects often use ASan in CI. valgrind is still better for: programs you can't recompile, kernel-style debugging, and detecting some advanced races. For CS 240, valgrind is what's available.
Reading valgrind output — what each part means
==12345== Invalid read of size 4
==12345==    at 0x4005A2: main (file.c:12)
==12345==  Address 0x520ABCD is 0 bytes after a block of size 40 alloc'd
==12345==    at 0x4C2BB8F: malloc (vg_replace_malloc.c:299)
==12345==    by 0x40058A: main (file.c:8)
  • "Invalid read of size 4" — you tried to read 4 bytes from forbidden memory
  • "at 0x... main (file.c:12)" — at line 12 of file.c (in main), the read happened
  • "is 0 bytes after a block of size 40" — you went exactly past the end of a 40-byte block
  • "alloc'd at file.c:8" — the block was malloc'd at line 8
This means: you malloc'd 40 bytes (10 ints), then read arr[10] — one past the end.
Memory leaks vs definite leaks vs possibly lost
valgrind classifies leaks into 4 categories at exit:
  • definitely lost — no pointer reaches this block (real leak, fix it)
  • indirectly lost — a leaked block's contents include pointers to other blocks
  • possibly lost — a pointer to the middle of the block exists; could be a leak or could be intentional
  • still reachable — pointers exist but you didn't free before exit; not technically a bug but bad style
The CS 240 custom malloc focuses on definite leaks and double-frees.
Why check valgrind even when your tests pass
A program can have memory bugs that don't manifest as visible failures. Reading uninitialized memory may "happen" to give the right answer this run. Buffer overruns may corrupt only memory that's about to be overwritten anyway. Then a TINY change (different optimization, different OS, different malloc implementation) suddenly exposes the bug in production. valgrind/ASan catches these latent bugs proactively.

1. Why doubly-linked?

A singly-linked list can only walk forward. To delete a node you need its predecessor — so you walk from head. Doubly-linked nodes keep a back-pointer too, allowing O(1) deletion given the node, plus reverse traversal.

struct double_l {
    int data;
    struct double_l *next_ptr;
    struct double_l *prev_ptr;
};
typedef struct double_l node_t;

2. Prepending to a doubly-linked list — order matters

void prepend(node_t **head_ref, int v) {
    node_t *n = malloc(sizeof(node_t));
    n->data = v;
    n->prev_ptr = NULL;        // 1) new node has no predecessor
    n->next_ptr = *head_ref;   // 2) point to old head
    if (*head_ref != NULL)
        (*head_ref)->prev_ptr = n; // 3) old head's prev = new node
    *head_ref = n;             // 4) head = new node
}
Four ordered steps
1) new node's prev = NULL · 2) new node's next = old head · 3) old head's prev = new node · 4) head = new node.

3. Removal from middle

Given a node n in the list, remove it in O(1):

void remove_node(node_t **head_ref, node_t *n) {
    if (n->prev_ptr) n->prev_ptr->next_ptr = n->next_ptr;
    else             *head_ref = n->next_ptr;     // was head
    if (n->next_ptr) n->next_ptr->prev_ptr = n->prev_ptr;
    free(n);
}

4. Detecting bugs — valgrind

valgrind is a tool that runs your program and reports memory errors: leaks, invalid reads/writes, double-frees, use-after-free.

$ gcc -g my_prog.c -o my_prog
$ valgrind --leak-check=full ./my_prog

Reports lines like:

==12345== Invalid read of size 4
==12345==    at 0x4005A2: main (my_prog.c:12)
==12345==  Address 0x520... is 0 bytes after a block of size 40 alloc'd
==12345==
==12345== HEAP SUMMARY:
==12345==     in use at exit: 40 bytes in 1 blocks
==12345== LEAK SUMMARY:
==12345==    definitely lost: 40 bytes in 1 blocks

5. The CS240 custom malloc

For homework, the course provides a custom malloc/free that detects:

  • Memory leaks (allocations never freed)
  • Double-frees
  • Out-of-bounds writes (via canary bytes around your block)

The library reports problems at exit. Read the output carefully — point losses are typically tied to leaks/double-frees.

How to approach a doubly-linked list operation

Recipe
  1. Draw the BEFORE state with arrows for next and prev
  2. Draw the AFTER state
  3. List the pointer rewrites needed (usually 2-4)
  4. Write them in a safe order: don't lose any node before re-linking
  5. Handle edge cases: empty list, only-node, head, tail
  6. free the removed node, set the variable to NULL

LECTURE 15 · POINTERS TO POINTERS

Double-Pointers, Internal Pointers, the Many Faces of Zero

Beyond the Slides — Deep Dive

When you actually need T**
You need a double-pointer parameter when the function must change WHICH OBJECT the caller's pointer points to. Three common cases:
  • Setting a caller's pointer to NULL — e.g., my_close(FILE **)
  • Returning a freshly-allocated pointer — e.g., create_thing(thing_t **out) instead of returning the pointer
  • Modifying the head of a linked list — prepend/delete-head changes which node IS the head
If you're only modifying what the pointer POINTS AT (not the pointer's value), T* is enough. Rule of thumb: count the asterisks needed to reach the thing you're changing.
Why use T** instead of returning T*
Two equivalent designs for "create a new linked list and give it to the caller": 
// Design A: return value
node_t *create_list(int v) {
    node_t *n = malloc(sizeof(*n));
    if (!n) return NULL;
    n->data = v; n->next = NULL;
    return n;
}

// Design B: out-parameter
int create_list(int v, node_t **out) {
    *out = malloc(sizeof(node_t));
    if (!*out) return -1;
    (*out)->data = v; (*out)->next = NULL;
    return 0;
}
Design B leaves the return value free for an error code (POSIX style). Both are common in C codebases.
Internal pointer struct = "shallow vs deep" copy problem
When a struct contains pointers, struct assignment (b = a) copies the pointer values — both structs now point to the same memory. This is a SHALLOW copy. Consequences:
  • Modifying through a->name also affects b->name
  • free(a->name) followed by free(b->name) is a double-free
  • If a->name is later reallocated to a new buffer, b->name dangles
For a true "duplicate" you need a DEEP COPY: malloc new buffers and strcpy/memcpy the contents.
qsort uses void**? Sort of
qsort's signature is void qsort(void *base, size_t n, size_t size, int (*cmp)(const void *, const void *)). Inside qsort, when it compares two elements, it passes pointers to those elements — but the elements themselves might be pointers (if you're sorting an array of strings, i.e. char **). So inside YOUR comparator, the args are const void * that you cast to char ** and dereference: strcmp(*(char**)a, *(char**)b). This is the canonical "double pointer in real code" example.
