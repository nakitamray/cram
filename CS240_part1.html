LECTURE 02 · TOOLCHAIN

Compiling, Object Files, Linking, Executing

1. The Big Picture: The Build Pipeline

When you type gcc hello.c -o hello, gcc orchestrates a four-stage pipeline. Knowing each stage matters because errors at each stage mean different things.

hello.c → [PREPROCESSOR cpp] → hello.i → [COMPILER cc1] → hello.s → [ASSEMBLER as] → hello.o → [LINKER ld] → hello (executable)
StageToolInput → OutputWhat it does
1. Preprocesscpp.c → .iExpand #includes, #defines, strip comments, evaluate #ifdefs
2. Compilecc1.i → .sTranslate C to assembly for the target CPU
3. Assembleas.s → .oTranslate assembly to machine code (object file)
4. Linkld.o + libs → exeResolve symbols, combine objects into executable
Beyond the slides — see each stage
You can stop at any stage:
  • gcc -E hello.c → preprocessed source on stdout
  • gcc -S hello.c → produces hello.s assembly
  • gcc -c hello.c → produces hello.o object file
  • gcc hello.c → all the way through linking, produces a.out
This is why "compile errors" ≠ "linker errors". A compile error means the compiler couldn't translate your code (syntax, type mismatch). A linker error means the compile succeeded but a symbol wasn't found (e.g. you called sqrt but didn't link -lm).

2. Why C, Why This Matters

  • Created by Dennis Ritchie at Bell Labs, 1969-1973
  • Goal: write UNIX in a portable language (the original UNIX was nearly all assembly, so each new architecture meant rewriting it)
  • By using C, only ~5% of UNIX needed to stay assembly
  • Today: kernels (Linux/Windows/macOS), JVM, embedded firmware, drivers, fast libraries are all C
C standards — version names you'll see
  • K&R C — pre-1989, defined by the Kernighan & Ritchie book
  • C89/C90 — first ANSI/ISO standard
  • C99 — what this class targets; added bool, // comments, designated initializers, VLAs
  • C11, C18, C23 — newer revisions

5. Object Files — What's Actually Inside

An object file (.o) is incomplete machine code. It contains:

  • Code section (.text) — machine code for this module's functions
  • Data section (.data, .bss) — initialized and zero-initialized globals
  • Symbol table — names this object DEFINES (e.g. main, my_func) and names it NEEDS (e.g. printf, strlen)
  • Relocation info — where addresses need to be patched up at link time
Inspect with nm and objdump
$ nm hello.o
0000000000000000 T main          # T = defined here in text section
                 U printf         # U = undefined, needs to be linked

$ objdump -d hello.o              # disassemble
The "U printf" is the key: this object knows it needs printf, but doesn't have it. The linker satisfies the U entries by looking through other .o files and libraries.

6. Linking — Two Reasons to Care

Why split code across multiple .o files?

  1. Incremental compilation — change one file, only that file recompiles. With 50 .c files this saves enormous time
  2. Source privacy — distribute .o files without the source (CS 240 homework releases use this — you get .o files for the parts you don't write)
  3. Code reuse — share .o files across multiple programs (libraries)
What the linker does
  1. Read each .o file's symbol table
  2. Match every "U" (undefined) symbol to a "T" (defined) elsewhere
  3. Patch addresses (relocate) so calls and references work
  4. Combine all sections into one executable
  5. Add startup code (the _start stub that calls main)
Common linker errors
  • undefined reference to `sqrt' — forgot -lm
  • multiple definition of `foo' — same function defined in 2+ files (or in a header without static)
  • undefined reference to `main' — no main function anywhere

7. Function Prototypes — Why They Exist

The compiler processes one .c file at a time, so when file2.c calls a function from file1.c, the compiler needs to know its signature. Without a prototype, the compiler ASSUMES int return and unspecified args — then links anyway, producing silent corruption.

/* file1.c */
float calc(float a, float b) { return a * b; }

/* file2.c — WRONG: no prototype */
#include <stdio.h>
int main() {
    float r = calc(11.10, 3.0);  // compiler thinks calc returns int!
    printf("%f\n", r);              // junk output
}

/* file2.c — CORRECT */
#include <stdio.h>
float calc(float, float);          // the prototype (declaration)
int main() {
    float r = calc(11.10, 3.0);
    printf("%f\n", r);
}

A header file (.h) is just a place to put prototypes (and struct/typedef declarations) so multiple .c files can share them without duplicating.

8. Execution — what ./ Means

$ ./hello

The ./ tells the shell "run the file 'hello' in the current directory" — by default the shell only searches PATH, not .. This is a security measure: if . were on PATH, an attacker could put a malicious ls in your directory.

What happens when you press Enter
  1. Shell fork()s a child process
  2. Child execve()s the executable
  3. Kernel reads the executable's headers (ELF format on Linux)
  4. Kernel maps code/data sections into the process's virtual address space
  5. Kernel jumps to the program's entry point (_start in libc)
  6. _start sets up argc/argv/envp on the stack and calls main()
  7. When main returns, _start calls exit() with main's return value

How to approach a "what gcc command" question

Recipe
  1. Identify the goal: object file (.o), executable, or just preprocessed output
  2. Object file → use -c
  3. Executable → omit -c, add -o name
  4. Always add -Wall -Werror -g for class assignments
  5. If using math: add -lm AT THE END (linker order matters)
  6. If multiple files: list them all; if any are .o, the linker handles them

LECTURE 03 · FILE I/O

Opening, Closing, Reading, Writing Files

1. The Mental Model — FILE * is a Handle

A FILE * is an opaque handle to a stream. You don't look inside it. You hand it to library functions which know how to use it.

What's actually inside FILE
FILE is a struct (defined in <stdio.h>) that wraps:
  • An OS file descriptor (an int — see Lecture 23)
  • A buffer (typically 4-8 KB) for performance
  • The current position
  • Error and EOF flags
  • Mode info (read/write/append, binary/text)
The buffering is why fclose matters: if your program exits without closing, buffered data may never reach the disk. stdout is line-buffered when connected to a terminal, fully-buffered when piped — that's why printf("hi"); for(;;); may show nothing on a terminal until you add \n.

Three FILE * are pre-opened at program start:

  • stdin — standard input (reads from keyboard / pipe)
  • stdout — standard output (terminal / pipe)
  • stderr — standard error (terminal, typically unbuffered)

Class rule — always check fopen
FILE *fp = fopen(path, "r");
if (fp == NULL) {
    perror("fopen");   // prints reason
    return -1;

Failing to check fopen's return is an instant point loss in this class.

3. fclose()

int fclose(FILE *fp);   // 0 on success, EOF on failure

fclose:

  1. Flushes the buffer (writes pending output)
  2. Releases the FILE struct memory
  3. Closes the underlying OS file descriptor

Crucially: fclose does NOT set your variable to NULL. After fclose, the pointer is dangling — using it is undefined behavior. Always:

fclose(fp);
fp = NULL;

5. fscanf — Formatted Input

int fscanf(FILE *fp, const char *fmt, ...);

// equivalent:
scanf("%d", &x);
fscanf(stdin, "%d", &x);

Returns the count of successful conversions, or EOF (-1) at end-of-file.

Why the &?

scanf needs to WRITE to your variable. C is pass-by-value, so passing x would only give scanf a copy. Instead you pass &x (the address) so scanf can deposit the result.

Common scanf pitfalls
  • Forgetting & on a non-array — segfault
  • Not checking the return — proceed with garbage if a conversion failed
  • Whitespace in format — %d skips leading whitespace; %c does not
  • %s reads until whitespace — won't capture multi-word strings; no length cap = buffer overflow
  • Mixing scanf and fgets — scanf often leaves a \n behind that fgets then "eats"

6. Complete pattern — read all ints from a file

#include <stdio.h>
int main(int argc, char *argv[]) {
    if (argc < 2) {
        fprintf(stderr, "usage: %s file\n", argv[0]);
        return 1;
    }
    FILE *fp = fopen(argv[1], "r");
    if (fp == NULL) {
        perror("fopen");
        return 1;
    }
    int n;
    while (fscanf(fp, "%d", &n) == 1) {
        printf("got %d\n", n);
    }
    fclose(fp);
    fp = NULL;
    return 0;
}

Recipe

  3. Check if (fp == NULL) return error;
  4. fprintf / fwrite the data; check return values for errors
  5. fclose(fp); fp = NULL;

argv, EOF Detection, scanf Sets, bool

1. main's Arguments — argc and argv

int main(int argc, char *argv[]);
// equivalent:
int main(int argc, char **argv);
  • argc — argument count, including the program name
  • argv — array of strings (each a char *)
  • argv[0] — program name as invoked
  • argv[argc] — guaranteed to be NULL (sentinel)
Example: ./prog hello world 42
argc = 4
argv[0] = "./prog"
argv[1] = "hello"
argv[2] = "world"
argv[3] = "42"      // always a string, even if it looks like a number
argv[4] = NULL
Beyond the slides — where argv lives
The shell creates the argv array on the program's stack before main runs. The strings themselves typically live just above the stack, with environment variables. argv strings are mutable (you can write into argv[1]) but going out of bounds wrecks the stack.
Converting argv to integer
#include <stdlib.h>
int n = atoi(argv[1]);   // no error reporting
long n2 = strtol(argv[1], NULL, 10);  // safer

2. EOF Detection — Why feof Is a Trap

Common mistake:

fscanf(fp, "%d", &n);
while (!feof(fp)) {
    printf("%d\n", n);
    fscanf(fp, "%d", &n);
}

Why this is wrong: feof returns true only AFTER a read attempt has hit EOF. So when the LAST fscanf returns EOF, the variable n is unchanged from the previous read — the loop processes the last value twice.

Correct pattern
while (fscanf(fp, "%d", &n) == 1) {
    printf("%d\n", n);
}

Use the function's RETURN VALUE to drive the loop. fscanf returns the count of successful conversions, or EOF (-1) when input ends before anything is read.

3. ferror and clearerr

int ferror(FILE *fp);   // non-zero if an error occurred
void clearerr(FILE *fp); // reset both error and EOF flags

Useful for distinguishing "EOF reached" (feof) from "I/O error" (ferror). In CS 240, prefer checking the return value of read/write operations.

4. scanset Format — %[]

%[chars] reads one or more characters matching the set:

FormatMeaning
%[abc]read a, b, or c
%[A-Z]read uppercase letters
%[0-9A-Za-z]read alphanumeric
%[^abc]read characters NOT a, b, c (^ = negate)
%[^\n]read up to (not including) newline = "rest of line"
Special characters in the set
  • To match ]: put it FIRST → %[]xyz]
  • To match -: put it LAST → %[xyz-]
ALWAYS use width
A buffer of size N can safely hold N-1 characters plus the '\0' terminator. Use width N-1 in the format: 
char buf[100];
fscanf(fp, "%99[^\n]", buf);   // reads up to 99, leaves room for \0
Without the width, scanf will happily write past the end of the buffer.

5. #include — Search Paths

#include <stdio.h>     // system path: /usr/include, /usr/local/include
#include "myheader.h"   // current dir first, then system path

Add directories with -I:

gcc -I/path/to/headers -I. file.c
What #include literally does
The preprocessor LITERALLY copies the header file's contents into your .c file at the location of the #include. There's no magic — the result is one giant translation unit. This is why a header with a function definition (not just declaration) causes "multiple definition" linker errors when included from 2+ .c files: each .c gets its own copy of the function.

6. bool — C99 Booleans

#include <stdbool.h>

bool ready = true;
if (!ready) { ... }
while (running) { ... }

bool is technically _Bool; true and false are macros. Old code (pre-C99) often defines its own:

typedef int bool;
#define TRUE 1
#define FALSE 0

For new code, use <stdbool.h>.

How to approach a "read every line of a file" question

Recipe
  1. Get filename (typically argv[1]); validate argc
  2. fopen with "r"; check NULL
  3. Allocate buffer of MAX size; the format width is MAX-1
  4. Loop: while (fscanf(fp, "%MAX_MINUS_1[^\n]\n", buf) == 1)
  5. Process the line
  6. fclose; ptr = NULL

LECTURE 05 · ASSERT & RANDOM ACCESS

The assert() Macro & fseek/ftell

1. Code Comments — When and Why

Bad comments restate the obvious
i = i + 1;  // add 1 to i  ← useless
// loop from 0 to 9
for (int i = 0; i < 10; i++) ...
Good comments explain WHY, not WHAT
// Skip header rows; the file format starts data at line 4.
for (int i = 0; i < 3; i++) read_line(fp);

// TODO: this only handles ASCII; needs UTF-8 support for issue #142
for (int i = 0; s[i] != '\0'; i++) ...

Mark incomplete code with TODO, FIXME, or XXX. Add comments when fixing bugs to explain the trap for future readers.

2. assert() — A Self-Checking Crash

#include <assert.h>
assert(condition);

If condition is true, nothing happens. If false, the program aborts immediately and prints something like:

prog: prog.c:42: foo: Assertion `ptr != NULL' failed.
What assert is — really
assert is a macro, not a function. It expands to roughly: 
#define assert(e) \
    ((e) ? (void)0 : __assert_fail(#e, __FILE__, __LINE__, __func__))
The #e turns the expression into a string for the error message. __assert_fail prints the message and calls abort(), which raises SIGABRT and triggers a core dump.

3. When to Use assert

USE assert for "this should never happen"
int divide(int a, int b) {
    assert(b != 0);  // caller's bug if it happens
    return a / b;
}

int *p = malloc(100);
assert(p != NULL); // for class — though malloc CAN fail in real code
DON'T use assert for runtime errors
// BAD — file might genuinely not exist
FILE *fp = fopen(path, "r");
assert(fp != NULL);

The right pattern is to handle missing files gracefully (return an error code, print a message). Asserts are for situations that indicate a programmer bug — invariants you trust at design time.

4. Disabling assertions — NDEBUG

$ gcc -DNDEBUG=1 -o prog prog.c

When NDEBUG is defined at compile time, every assert(...) expands to nothing. The expression is NOT evaluated.

Deadly mistake
// BAD — works in debug, breaks in release
assert(++counter > 0);
With NDEBUG defined, ++counter is never executed. The increment is silently lost. Rule: only put pure side-effect-free expressions inside assert.

5. Random-Access I/O — fseek and ftell

long ftell(FILE *fp);                        // current offset
int fseek(FILE *fp, long offset, int whence); // move
whenceMeaning
SEEK_SET (0)Offset from beginning
SEEK_CUR (1)Offset from current position
SEEK_END (2)Offset from end (negative offset = before end)

fseek returns 0 success, -1 on failure. ftell returns offset or -1.

Idioms
// rewind to start
fseek(fp, 0, SEEK_SET);
// or the dedicated function:
rewind(fp);

// jump to end
fseek(fp, 0, SEEK_END);

// get file size
fseek(fp, 0, SEEK_END);
long size = ftell(fp);
fseek(fp, 0, SEEK_SET);

// jump to byte 1024
fseek(fp, 1024, SEEK_SET);

// jump to record N (each record fixed size)
fseek(fp, N * sizeof(record_t), SEEK_SET);

// rewind 16 bytes
fseek(fp, -16, SEEK_CUR);
Why random access matters
For sequential text files, you read top to bottom. But for fixed-format binary records (like a database), you can jump straight to record N without reading the previous N-1. This is O(1) lookup. Combined with fwrite, you can build flat-file databases entirely in C.

How to approach a "jump to record N" question

Recipe
  1. Open the file in binary mode ("rb" or "rb+")
  2. Compute the offset: N * sizeof(record_t)
  3. fseek(fp, offset, SEEK_SET) — check return for -1
  4. fread the record; check that fread returned 1
  5. Use the data, optionally fseek back and fwrite to update
  6. fclose; ptr = NULL

LECTURE 06 · STRUCTURES

Structures, Declaration vs Definition, Strings

1. Why Structures Exist

You can group related data in arrays — but only if they're the same type. To represent a person with name, age, height, and a list of test scores, you need different types together. That's a struct.

C struct vs Java class — fundamental difference
A C struct is JUST DATA. There are no methods, no inheritance, no access modifiers, no constructors. Functions that operate on a struct are written separately and take the struct (or pointer) as a parameter. This is "data and behavior separated" — the opposite of OOP. C++ classes are basically structs with methods bolted on.

2. Assignment Suppression — %*

Insert * after % to scan and DISCARD a value:

fscanf(fp, "%d %*f %s", &n, name);
// reads int into n, reads & discards float, reads string into name

The skipped value isn't counted in the return value. Useful when a file has columns you don't need.

3. Declaring a Structure Type

struct point {
    int x;
    int y;
};   // the semicolon is REQUIRED

This declaration ALONE allocates no memory — it just defines a type. To create variables:

struct point a;              // uninitialized
struct point b = {3, 4};      // initialized
struct point c = {.x = 5};   // designated initializer (C99)

4. Declaration vs Definition — KEY DISTINCTION

TermEffectExample
DeclarationAnnounces type/signature, allocates no memorystruct point; or extern int x;
DefinitionAllocates storageint x; or int x = 5;
// Pure declaration (no var):
struct point {
    int x, y;
};

// Declaration AND definition (allocates 'origin'):
struct coord {
    int u, v;
} origin;

Function prototypes are declarations. extern turns a definition into a declaration ("this exists somewhere else"). Each variable must be DEFINED in exactly one .c file; can be DECLARED in many.

5. typedef — Type Aliases

struct point {
    int x, y;
};
typedef struct point point_t;

// now you can write:
point_t p1, p2;
// instead of:
struct point p3, p4;
One-shot typedef
typedef struct {
    int x, y;
} point_t;

This makes an anonymous struct and aliases it to point_t. The struct has no tag — you can ONLY refer to it by the typedef name.

6. Initializing Structs with Arrays

struct person {
    char name[40];
    char title[16];
    int codes[4];
};

struct person jt = {
    "Jeff Turkstra",
    "Troublemaker",

Nested braces for the inner array.

Assigning fields after declaration
#include <string.h>

7. Compound Literals (C99)

Re-assign an entire struct at once:

p = (struct person){"Bob", "VP", {1, 2, 3, 4}};

The cast-style syntax creates a temporary struct with those values and assigns it.

8. Struct Assignment Works

struct point a = {1, 2}, b;
b = a;   // MEMBERWISE COPY — works! Unlike Java references.
Shallow copy gotcha
Struct assignment copies the bytes. If a struct contains POINTERS, the copy points to the same memory: 
struct info { char *name; };
struct info a = { malloc(100) };
strcpy(a.name, "hi");
struct info b = a;        // SHARED pointer
b.name[0] = 'H';            // modifies a.name too!
free(b.name);             // later free(a.name) = double free

How to approach a "design a struct" question

Recipe
  1. List required fields with types and sizes
  2. Write struct name { fields };
  3. Add a typedef if you want to drop "struct" prefix later
  4. To create a variable: struct name var = {...}; with initializers in declaration order
  5. For string fields, init with literal or strncpy + force \0
  6. Access with var.field or ptr->field

LECTURE 07 · ARRAYS

Arrays, Initialization Quirks, the strncpy Trap

1. extern Recap

extern char g_loan_history[MAX][3][BUF];
extern int g_loan_count;

extern at file scope: declares a variable defined in another module. Doesn't allocate. The variable must be DEFINED in exactly one .c file (without extern).

2. The Auto-Initialization Rule

StorageInitial value (no explicit init)
Global (file scope)Zero (in BSS)
Static localZero
Plain local (auto)Garbage — whatever was on the stack
malloc'dGarbage
calloc'dZero
Where do globals get zeroed?
The OS, when it loads the program, allocates the BSS section and zeroes it. This happens BEFORE main runs. It's free — no per-byte writing — because the kernel can map it lazily from a zeroed page. This is why globals are "free to initialize to zero" but locals are "free to leave garbage."

3. Partial Initialization

int arr[200] = {1, 2, 3};
// arr[0]=1, arr[1]=2, arr[2]=3, arr[3..199] = 0

If you provide ANY initializer, all unspecified members are set to zero. Idiom for zeroing an array:

int arr[100] = {0};   // all 100 = 0
int arr[100] = {};      // C23 / GCC extension

4. Array Auto-Sizing

int a[] = {1, 1, 2, 3, 5, 8};   // size = 6

Compiler infers the size from the initializer count. Useful for tables that may grow.

How to compute count safely
int a[] = {10, 20, 30};
int n = sizeof(a) / sizeof(a[0]);   // n = 3

This works ONLY when the array is in scope as an array. Once passed to a function (where it decays to a pointer), sizeof returns pointer size.

5. Arrays of Structures

struct person people[3] = {
    {"Alice", "TA",  {1, 2, 3, 4}},
    {"Bob",   "TA",  {5, 6, 7, 8}},
    {"Cleo",  "Pf",  {9, 8, 7, 6}},
};

Each inner brace is one struct's worth of fields. Trailing comma after last element is allowed.

6. The strncpy Trap

char *strncpy(char *dest, const char *src, size_t n);

strncpy copies AT MOST n bytes from src to dest:

  • If strlen(src) < n: dest is null-terminated AND PADDED WITH ZEROS up to n bytes
  • If strlen(src) >= n: dest gets exactly n bytes, NO null terminator
The trap
char buf[5];
strncpy(buf, "Hello", 5);
printf("%s\n", buf);   // READS PAST END until random \0
"Hello" is 5 chars; strncpy copies all 5, leaving NO room for '\0'. printf with %s walks past buf looking for a terminator. Result: junk output and possibly a segfault.
Correct idiom
strncpy(buf, src, sizeof(buf) - 1);
buf[sizeof(buf) - 1] = '\0';
// or, equivalently:
strncpy(buf, src, sizeof(buf));
buf[sizeof(buf) - 1] = '\0';

Always force a '\0' as the last byte after strncpy.

7. Variable-Length Arrays — Disallowed in This Class

void bad(int n) {
    int arr[n];   // VLA — C99 added, but Turkstra forbids in CS 240
}

VLAs allocate on the stack with a runtime-known size. Stack-overflow risk and not portable to C++/older compilers. Use malloc instead.

How to approach an "init array of structs" question

Recipe
  1. Define the struct type
  2. Declare the array: struct T arr[N];
  3. Initialize with nested-brace literal: { {...}, {...}, ... }
  4. Each inner brace = one element's fields in declaration order
  5. For unset elements: globals = 0, locals = garbage (init explicitly)
  6. For string fields: use literal in initializer or strncpy + force \0 after

LECTURE 08 · MEMORY LAYOUT

How Data Sits in Memory: Padding, Alignment, Binary I/O

1. Memory as Bytes

Memory is a long row of bytes — addresses 0 to 2^N - 1 (N = pointer width: 32 or 64). Each byte is a value 0-255 or one ASCII character.

The process address space (Linux/x86_64)
0xFFFFFFFFFFFFFFFF  ┌──────────────────┐
                    │ kernel space     │
0x00007FFFFFFFFFFF  ├──────────────────┤
                    │ stack ↓          │  grows DOWN
                    │  (mmap region)   │  shared libs
                    │ heap ↑           │  grows UP from malloc
                    ├──────────────────┤
                    │ BSS              │  zero-initialized globals
                    │ data             │  initialized globals
                    │ text             │  your code
0x0000000000400000  └──────────────────┘
This is why char *p = "hi"; is read-only (literal lives in text/rodata) and malloc'd data lives in the heap.

Use sizeof instead of hardcoding numbers — your code stays portable.

3. Alignment — Why Padding Exists

Most CPUs require multi-byte values to be at addresses divisible by their size:

  • An int (4 bytes) wants address divisible by 4
  • A short (2 bytes) wants address divisible by 2
  • A double (8 bytes) wants address divisible by 8

If unaligned: some architectures (Sparc, old ARM) crash; x86 just slows down.

Padding within a struct
struct bad {
    char c;     // 1 byte at offset 0
    // 3 bytes of padding so int starts on multiple of 4
    int i;      // 4 bytes at offset 4
};
// sizeof = 8, not 5
Trailing padding for arrays
struct tail {
    int i;      // offset 0
    char c;     // offset 4
    // 3 bytes trailing padding
};
// sizeof = 8 — so arr[1] keeps int aligned

4. Reorder to Minimize Size

Order fields LARGEST to SMALLEST: doubles → long long → pointers → long → int / float → short → char. Result: padding only at the end (if any).

// BAD — 28 bytes on 32-bit
struct b { char c1; int i1; char c2; int i2;
            char c3; int i3; char c4; };

// GOOD — 16 bytes
struct g { int i1, i2, i3; char c1, c2, c3, c4; };

5. Binary I/O — fread & fwrite

size_t fwrite(const void *p, size_t sz, size_t n, FILE *fp);
size_t fread (void       *p, size_t sz, size_t n, FILE *fp);

Dump structures as raw bytes. Returns count of items (NOT bytes) transferred.

struct rec r = {42, 3.14};
FILE *fp = fopen("data.bin", "wb");
fwrite(&r, sizeof(r), 1, fp);
fclose(fp);

// Reading multiple records
while (fread(&r, sizeof(r), 1, fp) == 1) {
    process(&r);
}
Binary files NOT portable
A file written on x86 (little-endian, 64-bit, specific padding) may be unreadable on Sparc (big-endian, different alignment).

How to compute struct size

Recipe
  1. Walk fields top-to-bottom, tracking byte offset
  2. For each field, note required alignment (= its size for primitives)
  3. If current offset isn't a multiple of that alignment, ADD PADDING
  4. Place the field; advance offset by its size
  5. At the end, ROUND UP total to multiple of largest alignment in the struct

1. Bitfields — Sub-Byte Struct Fields

struct ieee_float {

    unsigned int exponent : 8;
    unsigned int mantissa : 23;

Each field uses exactly the specified number of bits, packed into the underlying int. Useful for hardware registers, but don't use bitfields in CS 240 homework — shown for awareness.

union data {
    int i;
    float f;
    char bytes[4];
};
// sizeof = 4 (largest member)

x |= (1 << N);    // SET bit N
x &= ~(1 << N);   // CLEAR bit N

if (x & (1 << N))  // TEST bit N (non-zero if set)
Combine flags
#define READ   (1 << 0)

#define EXEC   (1 << 2)

int perms = READ | WRITE;       // 0011

Recipe

  2. SET: x |= mask
  3. CLEAR: x &= ~mask
  4. TOGGLE: x ^= mask
  5. TEST: if (x & mask)

LECTURE 10 · POINTERS

Introduction to Pointers

The K&R Disclaimer

"Pointers have been lumped with the goto statement as a marvelous way to create impossible-to-understand programs. With discipline, however, pointers can be used to achieve clarity and simplicity."

The CONCEPT is simple: a pointer is a variable holding the address of another variable. Combinations and applications get tricky.

1. The Big Idea

Every variable has an address (its location in memory) and a value (data at that location). A POINTER is a variable whose value IS another variable's address.

int x = 42;       // suppose x is at address 0x1000
int *p = &x;     // p holds the value 0x1000
                  // p itself lives elsewhere, suppose at 0x2000

printf("%d\n", x);   // 42
printf("%p\n", &x);  // 0x1000
printf("%d\n", *p);  // 42 (deref: value at p's address)
printf("%p\n", p);   // 0x1000 (p's value = x's address)
printf("%p\n", &p);  // 0x2000 (p's own address)

2. The Two Operators

OperatorMeaningExample
&Address-of&x = "address of x"
* (in expression)Dereference*p = "the thing p points to"
* (in declaration)"is a pointer to"int *p; = "p is pointer to int"
Same symbol, different meanings
int *p;       // declaration: * means "p is a pointer"
*p = 5;       // expression: * means "dereference p"
This trips up beginners. Read declarations as "pointer to int", expressions as "dereference."

3. Defining a Pointer

int *p;          // uninitialized — DANGEROUS
int *p = NULL;   // initialized to "nowhere"
int x;
int *q = &x;     // initialized to point to x

An uninitialized pointer holds a random address. Dereferencing it crashes (lucky) or silently corrupts memory (unlucky).

Multiple declarations
int *p, q;     // p is int*, q is int (NOT int*)
int *p, *q;    // both are int*

4. Why Pointers — Pass-By-Reference

This DOESN'T work
void increment(int n) {
    n = n + 1;
}
int main() {
    int x = 0;
    increment(x);
    printf("%d\n", x);   // still 0!
}

C is pass-by-VALUE — increment got a COPY of x.

This works — pass-by-pointer
void increment(int *p) {
    *p = *p + 1;
}
int main() {
    int x = 0;
    increment(&x);
    printf("%d\n", x);   // now 1
}

The function still got a COPY of the pointer — but both pointers point to the SAME int, so dereferencing modifies the original.

5. NOW You Understand scanf

scanf("%d", &x);
// scanf needs to write into x
// it can't receive x's value; it needs x's address

For arrays, no & needed: an array name evaluates to its first element's address. scanf("%s", buf) works because buf ≡ &buf[0].

6. NULL — The Special Pointer Value

int *p = NULL;
if (p == NULL) ...    // "p doesn't point anywhere"
if (!p)         ...    // equivalent shorthand

NULL is a macro typically defined as ((void *)0). Dereferencing NULL is undefined — typically a segfault on Linux.

7. Pointer Sizes

All pointers, regardless of what they point to, have the SAME size on a given architecture:

  • 32-bit system: 4 bytes
  • 64-bit system: 8 bytes

char *, int *, struct foo *, void * — all 8 bytes on x86_64.

How to make a function modify the caller's variable

Recipe
  1. Change parameter type from T to T *
  2. Inside, dereference: read *p, write *p = ...
  3. Caller passes the address: f(&x)
  4. For arrays, name is already an address — pass directly
  5. Always check NULL before dereferencing
  6. Initialize all pointers (NULL or real address)

LECTURE 11 · MORE POINTERS

Pointer Arithmetic, Arrays, the Arrow Operator, gdb

1. Definition vs Indirection — Same Symbol, Different Meaning

int x = 7;
int *p = &x;   // this * declares p as a pointer
*p = 5;       // this * dereferences p

The * in declaration is part of the type. The * in an expression is the dereference operator. Context tells you which.

2. Pointers Behave Like Arrays

int arr[5] = {10, 20, 30, 40, 50};
int *p = arr;   // p points to first element

// All four are EQUIVALENT:
arr[2]
*(arr + 2)
p[2]
*(p + 2)

This is why arr[i] and *(arr + i) are interchangeable. Even i[arr] works (though never write it that way) — array subscript is just *(a + b) with operands commutative.

3. Arrays Decay to Pointers

An array name in most expressions automatically converts to a pointer to its first element. Three exceptions where the array stays an array:

  • sizeof(arr) — gives total bytes of the array
  • &arr — gives a pointer to the WHOLE array (different type from &arr[0])
  • String literal initializing a char[] — char s[] = "hi"; initializes the array directly
Differences between arrays and pointers
int arr[100];   // allocates space for 100 ints (400 bytes)
int *p;          // allocates space for ONE pointer (8 bytes on 64-bit)

p = arr;        // OK: pointer can hold an address
arr = p;        // COMPILE ERROR — array names are not assignable

4. Arrays as Function Parameters — A Lie

int sum(int arr[100]) {  // the [100] is ignored!
    // inside, arr is just int*
    printf("%zu\n", sizeof(arr));   // prints 8 (pointer size), not 400
}

Function parameters declared as arrays are SILENTLY rewritten to pointers. So you must pass the size separately:

int sum(int *arr, int n) {
    int total = 0;
    for (int i = 0; i < n; i++) total += arr[i];
    return total;
}

5. Pointer Arithmetic — In Element Units

int arr[10];
int *p = arr;
p++;            // advances by sizeof(int) bytes — 4, not 1
p = p + 3;     // advances by 3 * sizeof(int) = 12 bytes
p--;            // back by sizeof(int)

The compiler scales by the pointed-to type's size. This is why arr[i] works regardless of element size.

Pointer subtraction
int arr[100];
int *p1 = &arr[10];
int *p2 = &arr[20];
long diff = p2 - p1;   // = 10, NOT 40 bytes

Pointer subtraction returns the count of ELEMENTS between them.

6. Pointers and Structs — the Arrow Operator

struct point { int x, y; };
struct point pt = {3, 4};
struct point *pp = &pt;

// To access x via the pointer:
(*pp).x = 10;     // works but ugly
pp->x = 10;        // equivalent — preferred
Parens matter
*pp.x would parse as *(pp.x) — try to use pp's x field as a pointer. Wrong! Use (*pp).x or pp->x. The arrow exists precisely to avoid this paren mistake.

Stack vs Heap, malloc / calloc / free

1. The Stack-Local Pointer Trap

You can't return an array from a C function by returning a pointer to a local variable. The local lives on the stack, which is destroyed when the function returns.

/* DO NOT DO THIS */
int *make_array() {
    int arr[10] = {0};
    return arr;             // dangling pointer!
}
Why exactly is this broken?
When make_array returns, the stack frame containing arr is popped. The next function call REUSES that stack space — overwriting your data. The pointer returned is "dangling": it points to memory you no longer own. Reading or writing through it is undefined behavior.

The fix: allocate on the HEAP with malloc. Heap memory persists until you free it.

2. Stack vs Heap — What's the Difference

AspectStackHeap
AllocationAutomatic (function entry)Manual (malloc)

How malloc works under the hood
malloc is a USER-SPACE LIBRARY function. It maintains a pool of memory obtained from the OS via system calls (typically brk/sbrk for small allocations, mmap for large ones). Internally it manages "free lists" of available blocks, tries to reuse freed memory, and only asks the OS for more when needed. Each malloc'd block has a small HEADER (typically 8-16 bytes) recording its size — that's how free knows how much to release without you telling it.

3. malloc — Manual Allocation

void *malloc(size_t size);
  • size_t is an unsigned integer type (typically unsigned long)
  • Returns void * on success — implicitly converts to any pointer type in C
  • Returns NULL on failure (out of memory)
  • Memory is uninitialized — contains garbage
  • Lives on the heap until free()
#include <stdlib.h>
int *p = malloc(10 * sizeof(int));   // 10 ints (40 bytes on most systems)
if (p == NULL) {
    fprintf(stderr, "out of memory\n");
    return -1;
}
p[0] = 42;
// ... use p ...
free(p);
p = NULL;
Idiom — sizeof(*p) is more robust
int *p = malloc(N * sizeof(*p));   // not sizeof(int)
If you later change p's type from int * to long *, the malloc call automatically updates. Less to keep in sync.

4. calloc — Allocate AND Zero

void *calloc(size_t count, size_t size);
  • Allocates count * size bytes
  • Zeros every byte — unlike malloc
  • Two arguments, not one (count and per-element size)
  • Returns NULL on failure
int *p = calloc(10, sizeof(int));   // 10 zeroed ints
When to use calloc vs malloc + memset

Use calloc when you need zeros from the start. It's typically faster than malloc + memset because the OS can hand you pre-zeroed memory pages.

5. free — Return Memory to the Heap

void free(void *ptr);
  • Releases the block back to the heap allocator
  • free(NULL) is safe — defined as a no-op
  • free does NOT set your pointer to NULL — you must do it
  • Double-free = undefined behavior (crash or corruption)
  • Use-after-free = read/write to memory you don't own
  • free on a non-malloc'd pointer = undefined
free(p);
p = NULL;   // always do this
Why set to NULL after free?
If you accidentally use p later, dereferencing NULL crashes IMMEDIATELY (clear bug). Without setting to NULL, p still holds the old (now-invalid) address — using it might silently work for a while (the memory hasn't been reused yet), then suddenly corrupt random data when the allocator reuses that block. Crash early, debug easily.

6. Memory Leaks

If you malloc but never free, that memory stays allocated until the program exits. In a long-running program, leaks accumulate and the program eventually runs out of memory.

while (1) {
    int *p = malloc(1000);   // LEAK every iteration
    process(p);
    // missing: free(p);
}   // process slowly grows in memory until OOM

Tools like valgrind (Lecture 14) and the CS 240 custom malloc detect leaks at exit.
